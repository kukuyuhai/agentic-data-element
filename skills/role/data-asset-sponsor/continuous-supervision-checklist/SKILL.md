# 🔍 continuous-supervision-checklist

数据资产持续督导清单。在数据资源入表后，保荐机构（`dam-09`）需按**季度**或**年度**定期回顾数据资产的状态变化，监控使用量、收益实现、减值迹象及纠错事项。依据财会〔2023〕11号第六条（减值迹象判断）和第七条（后续计量要求），及时发现并提示企业做出调整。

## When to Use

- 数据资产入表后，保荐机构需按季度/年度执行持续督导。
- 发现数据资产的业务使用量出现**显著下降**时，触发专项督导。
- 市场或技术环境发生**重大不利变化**，可能影响数据资产价值时。
- 企业发生数据安全事件或合规问题后，需要评估对数据资产的影响。
- 企业拟对已有的数据资产会计处理进行**更正或变更**。

## 工作原则

1. **定期不遗漏**：无论是否有异常，入表后每个报告期至少执行一次督导。
2. **比例导向**：关注变动幅度而非绝对值，对**超过20%的变动**进行深入分析。
3. **触发即行动**：减值迹象触发后，应及时建议企业执行减值测试（见 `amortization-scheduler`）。
4. **纠错不回避**：发现前期差错时，不得以"不重要"为由回避，应量化影响后建议追溯重述。
5. **记录可追溯**：每次督导形成督导记录，留存作为下次督导的对比基准。

## 一、使用量监控

监控数据资产在报告期的实际使用情况，判断是否存在使用量大幅下降带来的减值风险。

### 监控指标

| 指标 | 季度关注 | 年度关注 | 数据来源 |
|------|----------|----------|----------|
| API 调用次数 | 环比变化 > ±30% | 同比变化 > ±50% | API 网关日志/数据服务统计 |
| 活跃用户数 | 环比变化 > ±20% | 同比变化 > ±30% | 业务系统/用户中心 |
| 数据产品访问频次 | 环比变化 > ±25% | 同比变化 > ±40% | 数据产品后台统计 |
| 数据更新频率 | 低于设计频率的 70% | 低于设计频率的 50% | ETL 任务调度系统 |
| 下游系统依赖度 | 依赖方退出 > 20% | 依赖方退出 > 30% | 系统间依赖关系矩阵 |

### 异常判断阈值

- **注意**：单项指标连续两个季度同向变动且累计超过30%。
- **预警**：多项指标（≥3项）同步恶化。
- **触发减值测试**：使用量低于入表时预测基准的50%。

## 二、收益实现监控

对照入表时的经济收益预期，监控实际收益的达成情况。

### 监控指标

| 指标 | 计算公式 | 关注阈值 | 数据来源 |
|------|----------|----------|----------|
| 收入达成率 | 实际收入 / 预期收入 × 100% | < 80% | 收入明细账/合同系统 |
| 成本回收率 | 累计已回收收入 / 入表成本 × 100% | < 60%（第2年末） | 收入明细账 vs. 账面原值 |
| 毛利率 | (收入 - 直接成本) / 收入 × 100% | < 行业均值 - 10pp | 收入成本报表 |
| ARPU（单位用户收入） | 收入 / 活跃用户数 | 环比下降 > 15% | 收入账单 + 活跃用户统计 |
| 合同续约率 | 续约合同数 / 到期合同数 × 100% | < 75% | CRM 系统 |

### 异常处理

- 收入达成率低于60%且无明显改善趋势 → 建议重新评估减值迹象。
- 成本回收率远低于预期 → 建议重新评估收益期限和估值假设。
- 毛利率持续下降 → 建议分析成本结构变化，评估存货跌价。

## 三、减值迹象监控

依据财会〔2023〕11号第六条，逐项监控减值迹象。

### 减值迹象清单

| 序号 | 减值迹象 | 判断标准 | 触发后行动 | 频率 |
|------|----------|----------|------------|------|
| IS-1 | 技术过时 | 处理技术/数据格式被替代或淘汰 | 建议执行减值测试，评估可收回金额 | 季度 |
| IS-2 | 市场萎缩 | 同类数据产品的交易价格/成交量持续下降（连续2季） | 建议执行减值测试，获取市场案例 | 季度 |
| IS-3 | 法律变化 | 新法规使得数据采集/使用/交易受限 | 立即建议法律合规评审 + 减值测试 | 按需 |
| IS-4 | 使用量大幅下降 | 使用量低于入表时预测基准的50% | 建议立即执行减值测试 | 季度 |
| IS-5 | 资产闲置 | 连续一个季度无使用记录 | 建议评估是否已丧失使用价值 | 季度 |
| IS-6 | 数据完整性受损 | 数据丢失/篡改/损坏超过10% | 建议评估修复成本与减值金额 | 按需 |
| IS-7 | 关键客户流失 | 前3大客户流失合计占比 > 40% | 建议重新评估收益预期和减值 | 季度 |
| IS-8 | 安全事件 | 发生数据泄露/违规，影响数据可用性或合规性 | 建议应急评估 + 减值测试 | 按需 |
| IS-9 | 经营环境变化 | 企业主营业务转型，现有数据不再相关 | 建议评估数据资产的使用价值或处置 | 按需 |

### 减值测试触发条件

满足以下**任一条件**，必须执行减值测试：

1. 上述 IS-1 至 IS-9 中**任意一条**被触发。
2. 两条及以上**注意**级别及以上异常同时存在。
3. 财务主管或保荐人认为有必要。

执行减值测试时，调用 `amortization-scheduler` 的减值测试模块。

## 四、纠错事项

监控入表后的前期差错、会计估计变更、以及需要重分类的事项。

### 纠错事项清单

| 序号 | 纠错类型 | 常见示例 | 处理方式 | 频率 |
|------|----------|----------|----------|------|
| CE-1 | 成本归集差错 | 将非资产化支出错误归入数据资产成本 | 追溯重述前期报表 | 按需 |
| CE-2 | 分类错误 | 将应列作存货的数据错误列作无形资产 | 重分类调整 | 按需 |
| CE-3 | 摊销年限变更 | 因技术迭代导致实际可使用年限短于原估计 | 会计估计变更（未来适用法） | 按需 |
| CE-4 | 减值准备补提 | 前期未识别减值迹象、未计提减值 | 追溯重述（如属前期差错）或当期确认 | 按需 |
| CE-5 | 资本化时点错误 | 提前或延后将研发支出资本化 | 追溯重述前期报表 | 按需 |

### 纠错流程

1. **识别**：在督导过程中或企业自查中发现纠错事项。
2. **量化**：计算差错的影响金额及对当期和前期报表的影响。
3. **定性**：判断属于**前期差错**（追溯重述）还是**会计估计变更**（未来适用）。
4. **建议**：向企业会计部门提出处理建议。
5. **追踪**：监督企业是否按要求完成更正。

## Inputs

| 字段 | 类型 | 必填 | 说明 | 示例 |
|------|------|------|------|------|
| `supervision_period` | string | 是 | 督导期间 | `"2025-Q4"` / `"2025年度"` |
| `previous_checklist` | object | 否 | 上期督导清单（首次督导可不提供） | 上期输出结果 |
| `usage_data` | object | 是 | 使用量数据 | 含 API调用次数、活跃用户数等 |
| `revenue_data` | object | 是 | 收入数据 | 含收入达成率、成本回收率等 |
| `market_data` | object | 否 | 市场环境数据 | 含同类数据产品价格趋势等 |
| `legal_changes` | array | 否 | 法律法规变化清单 | 本期新发布或修订的法规 |
| `security_incidents` | array | 否 | 安全事件报告 | 本期发生的安全事件 |
| `accounting_events` | array | 否 | 本期发生的会计事项 | 含分类调整、摊销变更等 |

## Outputs

| 字段 | 类型 | 说明 |
|------|------|------|
| `checklist_report` | object | 完整持续督导报告 |
| `usage_assessment` | object | 使用量评估（各项指标的实际/变化/评估） |
| `revenue_assessment` | object | 收益实现评估 |
| `impairment_indicators` | array | 触发的减值迹象清单（含触发时间、证据） |
| `impairment_test_required` | boolean | 是否需要执行减值测试 |
| `error_corrections` | array | 本期识别的纠错事项 |
| `recommendations` | array | 督导建议（含行动项、责任人、期限） |
| `next_supervision_date` | string | 下一次督导计划日期 |

## Steps

1. **获取上期基准**：若为首次督导，以入表时点为基准；若为期后督导，载入上期督导清单。
2. **收集使用量数据**：从 API 网关/业务系统获取本期使用量数据，计算各指标及变化幅度。
3. **评估使用量**：逐项判断是否触发异常阈值，将触发的指标录入 usage_assessment。
4. **收集收益数据**：从收入系统获取本期收益数据，计算达成率/回收率/毛利率等。
5. **评估收益实现**：与入表预期收益或上期收益做对比，识别大幅偏差。
6. **逐项检查减值迹象**：对照 IS-1 至 IS-9 清单，逐项检查是否触发，录入 impairment_indicators。
7. **判断减值测试必要性**：若有减值迹象触发，设置 impairment_test_required = true。
8. **检查纠错事项**：审查本期会计处理变化，判断是否有需要追溯重述或重新分类的事项。
9. **形成督导建议**：将评估结果转化为具体的行动建议。
10. **编制督导报告**：整合为完整的 checklist_report，标注下一次督导日期。
11. **标注标准溯源**：每条判断末尾标注引用的准则条款（如 `§第六条` / `§第七条`）。

## 常见问法与处理策略

| 问法 | 处理策略 | 注意事项 |
|------|----------|----------|
| "使用量下降很快，但企业说下个月会好" | 以事实数据为准，不以后续预期为由推迟减值判断；持续监控，设置宽限期 | 宽限期不超过一个季度，逾期未回升即建议减值测试 |
| "是不是每次督导都要做减值测试" | 不一定，仅当减值迹象被触发时才执行完整减值测试；常规督导可只做迹象筛查 | 不做测试 ≠ 不关注，每次督导仍须逐项筛查迹象 |
| "企业想修改摊销年限" | 引导企业提供年限变更的证据依据（技术评估报告/行业对标），按会计估计变更处理 | 变更摊销年限属会计估计变更，采用未来适用法 |
| "前期差错和会计估计变更怎么区分" | 前期差错是事实性错误（如成本漏记），需追溯重述；估计变更是基于新信息的判断变化，采取未来适用法 | 关键区别：事实性错误 vs. 判断性变化 |
| "季度督导和年度督导有什么不同" | 季度督导侧重使用量/快速指标，年度督导增加全面收益评估、减值测试深度审查和完整披露质量复检 | 季度轻量、年度全面 |
