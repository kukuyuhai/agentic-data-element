# 🛡️ security-audit-checklist

## When to Use

当需要进行数据安全相关审计或合规评估时启用本技能：

- 年度数据安全审计：按《数据安全法》要求定期开展数据安全审计；
- 数据安全合规检查：监管机构检查前或企业内部自查；
- 数据资产入表前的安全合规审查：确认数据资产的安全管理满足入表要求；
- ISO 27001 / 等保 / DSMM 等认证评估前的自检；
- 数据交易/合作前的安全尽调：向合作方证明自身数据安全管理水平；
- 安全事件后整改复查：确认整改措施已落实到位。

## 工作原则

1. **全面覆盖**：检查表覆盖数据安全管理的六大核心维度，不留死角。
2. **可量化**：每个检查项均有明确的通过标准（是/否/部分/不适用），审计结果可量化评分。
3. **溯源可查**：每个检查项均关联适用的法规标准条款，便于合规举证。
4. **分级处理**：审计发现按严重程度（严重/一般/建议）分级，便于整改优先级排序。
5. **闭环管理**：审计不是终点，必须配套整改建议和复查机制。

## 六大审计维度

### 维度一：数据分级分类

**法规依据**：`数据安全法（2021）第二十一条`——国家建立数据分类分级保护制度。

| 编号 | 检查项 | 通过标准 | 常见问题 |
|------|--------|---------|----------|
| CC-01 | 是否建立了数据分类分级制度？ | 有正式发布的分类分级管理办法或制度文件 | 制度未正式发布、仅口头约定、制度超过2年未更新 |
| CC-02 | 是否对所有数据资产完成了分类定级？ | 数据资产目录中每条资产均有分类和级别标注 | 仅对核心系统做了分级、存在未定级的数据孤岛 |
| CC-03 | 分类分级结果是否定期更新（至少每年）？ | 最近一次更新在12个月内 | 业务变化后分级未同步更新、新增数据未纳入 |
| CC-04 | 不同级别数据是否实施了差异化管控措施？ | 不同级别数据有对应的存储、访问、传输、销毁策略 | 仅做了分级但管控措施一刀切、高级别数据未加密 |
| CC-05 | 重要数据和核心数据是否已向主管部门报备？ | 有备案回执或主管部门确认函 | 自行认定重要数据但未报备、不清楚报备流程 |
| CC-06 | 数据分类分级结果是否在组织内传达培训？ | 有培训记录，涉及数据处理的岗位均参与 | 仅信息安全部门知晓、业务部门不了解数据级别 |

**严重问题（必须整改）**：CC-01 未建立制度、CC-02 未完成分类、CC-04 无差异化管控。

### 维度二：访问控制

**法规依据**：`GB/T 43697-2024`——数据安全能力成熟度模型中关于访问控制的要求。

| 编号 | 检查项 | 通过标准 | 常见问题 |
|------|--------|---------|----------|
| AC-01 | 是否建立了数据访问权限管理制度？ | 有权限申请、审批、授予、回收的完整流程文件 | 权限申请靠口头、无审批流、离职人员权限未回收 |
| AC-02 | 是否遵循最小权限原则？ | 账号权限按岗位所需的最小范围设置，定期审计无过度授权 | 默认给予宽泛权限、开发人员有生产数据全量访问权 |
| AC-03 | 是否有规范的权限矩阵并映射至角色？ | 有角色-资源-操作权限的矩阵文档，且与实际配置一致 | 权限矩阵停留在纸面、实际配置与文档不一致 |
| AC-04 | 是否实施了数据操作的审批流程？ | 敏感数据的高风险操作（导出、删除、批量查询）有审批流 | 批量导出敏感数据无需审批、审批流形同虚设 |
| AC-05 | 权限是否定期审计（至少每季度）？ | 最近一次权限审计在3个月内，有审计报告 | 从未做过权限审计、审计后发现问题未跟踪整改 |
| AC-06 | 是否实施了多因素认证（MFA）？ | 核心系统/高敏感系统的管理后台强制 MFA | 管理后台仅密码登录、MFA 仅部分人员启用 |

**严重问题（必须整改）**：AC-02 未遵循最小权限、AC-04 高风险操作无审批、AC-06 核心系统无 MFA。

### 维度三：审计日志

**法规依据**：`数据安全法（2021）第二十九条`——数据处理活动应留存日志。

| 编号 | 检查项 | 通过标准 | 常见问题 |
|------|--------|---------|----------|
| AL-01 | 是否全面记录了数据操作日志？ | 数据访问、修改、删除、导出、批量查询等关键操作均有完整日志 | 仅记录了系统登录日志、数据读取/导出无日志 |
| AL-02 | 日志记录是否包含必要字段？ | 日志包含操作人、时间、IP、操作类型、操作对象、操作结果 | 日志格式不规范、缺少关键字段（如操作对象） |
| AL-03 | 日志是否防篡改？ | 日志采用追加写入模式或有数字签名/哈希校验防止篡改 | 日志可被管理员直接删除、日志存储未做防篡改 |
| AL-04 | 日志保留期是否不低于6个月？ | 所有安全审计日志的保存期限 ≥ 6个月 | 日志定期被覆盖、存储空间不够导致旧日志丢失 |
| AL-05 | 是否配置了异常行为告警？ | 有基于规则的异常告警（如批量下载、非工作时间操作、异地登录） | 有日志但无告警配置、告警未及时响应 |
| AL-06 | 日志是否定期审查（至少每月）？ | 最近一次日志审查在1个月内，有审查报告 | 日志仅存储不审查、审查流于形式 |

**严重问题（必须整改）**：AL-01 关键操作无日志、AL-03 日志可篡改、AL-04 保留期不足6个月。

### 维度四：加密保护

**法规依据**：`数据安全法（2021）第二十七条`——数据处理者应采取加密等安全技术措施。

| 编号 | 检查项 | 通过标准 | 常见问题 |
|------|--------|---------|----------|
| EN-01 | 敏感数据是否实施存储加密？ | 个人敏感信息、重要数据及以上的存储已加密（AES-256/GM） | 敏感数据明文存储、仅数据库层面做了传输加密 |
| EN-02 | 数据传输是否使用安全协议？ | 内外部传输使用 TLS 1.2+ 或国密标准 | 内网明文传输、使用已废弃的 TLS 1.0/1.1 |
| EN-03 | 密钥管理是否规范？ | 有密钥生命周期管理（生成、分发、轮换、销毁），密钥与加密数据分离存储 | 密钥硬编码在代码中、密钥和加密数据存于同一服务器、无轮换策略 |
| EN-04 | 是否实施了数据脱敏展示？ | 应用的界面展示层面已对敏感字段脱敏（如手机号中间位隐藏） | 查询结果全量明文展示、前端的脱敏可绕过 |
| EN-05 | 加密算法是否合规？ | 使用符合国家密码管理要求的商密算法或国密算法 | 使用自研未验证加密算法、使用已知弱加密算法（如 DES/MD5） |
| EN-06 | 是否定期进行加密有效性验证？ | 至少每年一次渗透测试验证加密防护的有效性 | 部署了加密但从未验证是否真正生效 |

**严重问题（必须整改）**：EN-01 敏感数据明文存储、EN-03 密钥管理不规范、EN-05 使用弱加密算法。

### 维度五：备份恢复

**法规依据**：`GB/T 43697-2024`——数据安全能力成熟度模型中关于数据备份恢复的要求。

| 编号 | 检查项 | 通过标准 | 常见问题 |
|------|--------|---------|----------|
| BR-01 | 是否制定了数据备份策略？ | 有书面的备份策略，明确备份频率、周期、范围和保留期限 | 口头约定、策略不覆盖全部重要数据 |
| BR-02 | 备份策略是否与数据等级匹配？ | 不同级别数据有不同的备份频率和保留策略 | 所有数据一律日备、核心数据备份频率不足 |
| BR-03 | 是否定期执行恢复演练（至少每年）？ | 最近一次恢复演练在12个月内，有演练报告和改进措施 | 从未做过恢复演练、演练后问题未跟踪 |
| BR-04 | 备份数据是否加密存储？ | 备份介质上的数据已加密（敏感数据级） | 备份磁带/云存储未加密、备份传输明文 |
| BR-05 | 是否实施了异地备份/容灾？ | 备份数据存储于与生产环境不同的物理位置或可用区 | 仅本地备份、备份与生产在同一机房/机柜 |
| BR-06 | 备份恢复时间目标（RTO）是否定义并验证？ | 有明确的 RTO，且最近一次演练实际恢复时间达标 | RTO 仅纸面定义、实际恢复远超目标时间 |

**严重问题（必须整改）**：BR-03 从未进行恢复演练、BR-05 无异地备份、BR-01 无备份策略。

### 维度六：应急响应

**法规依据**：`数据安全法（2021）第二十九条`——发生数据安全事件时应立即采取处置措施并报告。

| 编号 | 检查项 | 通过标准 | 常见问题 |
|------|--------|---------|----------|
| IR-01 | 是否制定了数据安全应急预案？ | 有正式发布的数据安全事件应急预案或应急响应 playbook | 预案从未编写、预案已过期（超过2年未更新） |
| IR-02 | 是否明确定义了事件分级标准？ | 有明确的事件分级标准（P0/P1/P2），各级有对应的响应 SLA | 事件来了临时判断、无量化标准 |
| IR-03 | 是否定期开展应急演练（至少每年）？ | 最近一次演练在12个月内，含桌面推演或实战演练 | 从未演练、以真实事件代替演练 |
| IR-04 | 是否建立了安全事件通报机制？ | 有明确的内部上报流程和对外通报（监管/用户/公众）的模板和责任人 | 事件发生后不知道怎么通报、延误通报时间 |
| IR-05 | 应急联系人名录是否维护更新？ | 名录中包含责任人及备选人员的 7×24 联系方式，最近一次更新在3个月内 | 联系人已离职但名录未更新、缺少备选人员 |
| IR-06 | 是否在事件后开展了根因分析和整改？ | 安全事件后均有根因分析报告（RCA）和整改跟踪记录 | 事件解决即止、无根因分析、同类事件反复发生 |

**严重问题（必须整改）**：IR-01 无应急预案、IR-04 无通报机制、IR-03 从未演练。

## 审计检查表汇总模板

```markdown
## 数据安全审计检查表

| 审计基本信息 | |
|-------------|---|
| 审计日期 | YYYY-MM-DD |
| 审计范围 | 全部数据资产 / 核心系统 / 特定系统名称 |
| 审计标准 | 数据安全法（2021）、GB/T 43697-2024 |
| 审计人员 | 姓名 |
| 受审部门 | 部门名称 |

### 审计结果汇总

| 维度 | 总项 | 通过 | 未通过 | 不适用 | 合规率 |
|------|------|------|--------|--------|--------|
| 分级分类 | 6 | 5 | 1 | 0 | 83% |
| 访问控制 | 6 | 4 | 2 | 0 | 67% |
| 审计日志 | 6 | 6 | 0 | 0 | 100% |
| 加密保护 | 6 | 4 | 1 | 1 | 80% |
| 备份恢复 | 6 | 5 | 1 | 0 | 83% |
| 应急响应 | 6 | 3 | 3 | 0 | 50% |
| **合计** | **36** | **27** | **8** | **1** | **77%** |

### 严重问题清单

| 编号 | 维度 | 问题描述 | 整改建议 | 建议完成期限 |
|------|------|---------|----------|-------------|
| AC-02 | 访问控制 | 开发人员拥有生产数据库全量查询权限 | 回收权限，建立只读账号+审批流程 | 7天内 |
| ... | ... | ... | ... | ... |
```

## Inputs

| 字段 | 类型 | 必填 | 说明 | 示例 |
|------|------|------|------|------|
| `audit_scope` | string | 是 | 审计范围 | "全部数据资产" / "核心业务系统" |
| `audit_standard` | string | 是 | 适用的审计标准 | "数据安全法（2021）" / "GB/T 43697-2024" / "ISO 27001" |
| `audit_type` | string | 是 | 审计类型 | "年度审计" / "合规检查" / "自检" / "入表审查" |
| `prior_audit_result` | object | 否 | 上次审计的结果（用于跟踪整改） | `{date:"2024-06-01",compliance_rate:75,unresolved_issues:["AC-02","IR-03"]}` |
| `system_inventory` | list[object] | 否 | 被审计系统/资产清单 | `[{系统名称:"用户平台",数据类型:"个人+交易",级别:"重要"}]` |
| `focus_dimensions` | list[string] | 否 | 关注的维度（不指定则全维度审计） | `["访问控制","加密保护"]` |

## Outputs

| 字段 | 类型 | 说明 |
|------|------|------|
| `audit_report` | object | 完整审计报告：各维度检查结果、合规率、发现问题清单 |
| `compliance_rate` | number | 总体合规率（百分比） |
| `dimension_scores` | list[object] | 各维度合规率明细 |
| `critical_issues` | list[object] | 严重问题清单（必须整改项） |
| `general_issues` | list[object] | 一般问题和改进建议 |
| `remediation_plan` | list[object] | 整改计划（问题-建议-责任部门-完成期限） |
| `trend_analysis` | object | 与上次审计的对比分析（如有 prior_audit_result） |
| `next_audit_date` | string | 下次建议审计日期 |
| `next_actions` | list[object] | 后续行动建议 |

## Steps

**前置条件**：用户已明确审计范围和适用标准。

1. **确认审计参数**：
   - 确认审计范围（全部/指定系统）
   - 确认适用标准（数据安全法 / GB/T 43697-2024 / ISO 27001 等）
   - 如果提供了上次审计结果，加载未解决问题清单以跟踪整改情况
2. **加载检查项**：
   - 根据 `focus_dimensions` 选择需要审计的维度
   - 如果未指定，加载全部六大维度
3. **逐维度逐项审计**：
   - 对每个检查项，询问实际情况
   - 判定：通过 / 未通过 / 不适用
   - 对未通过项，分类为严重问题或一般问题
4. **计算合规率**：
   - 各维度合规率 = 该维度通过项数 / 该维度适用项数
   - 总合规率 = 总通过项数 / 总适用项数
5. **生成问题清单**：
   - 严重问题（必须整改）：标红，建议7-30天内完成
   - 一般问题（建议改进）：标黄，建议1-3个月内完成
   - 观察项：提示趋势性风险
6. **制定整改计划**：
   - 每个问题指定：整改措施、责任部门、建议完成期限
   - 标注依赖关系（部分整改需先完成前置项）
7. **趋势分析**（如有历史数据）：
   - 与上次审计对比，分析趋势（改善/恶化/持平）
   - 标注未整改到位的老问题
8. **建议下次审计日期**：
   - 通常建议12个月后，若合规率 < 70% 则建议6个月后
9. **输出完整审计报告**：
   - 按 Outputs 格式整理
   - 当涉及应急响应维度且发现问题时，建议引用 `incident-response-playbook`

## 常见问法与处理策略

| 问法 | 处理策略 |
|------|----------|
| "我们没有分类分级制度，能跳过去吗？" | 不能跳过。分级分类是数据安全法的法定要求（第二十一条），是其他安全措施的基础。应标注为严重问题，并建议第一步就是建立制度。 |
| "有些检查项我们确实做不到怎么办？" | 如实标注"未通过"，在整改计划中注明客观困难和替代措施。例如无法异地备份时，可考虑加强本地备份频率和存储可靠性作为风险缓释。但核心项无法满足的应如实向监管说明。 |
| "审计结果不好会不会有法律风险？" | 内部自检的审计结果不会主动产生法律风险，但需要通过整改降低实际风险。如果监管检查时审计记录显示长期未整改的严重问题，可能构成"未履行数据安全保护义务"的违规认定。 |
| "能用这个检查表替代第三方审计吗？" | 不能完全替代。本检查表适用于内部自查和准备性评估。对于监管要求的审计或认证评估，通常需要具备资质的第三方审计机构出具正式报告。 |
| "上次审计发现的问题还没改完怎么办？" | 在本次审计中标注为"遗留问题"，继续在整改计划中追踪。如果长期未整改且问题严重，这本身就是一个新的审计发现（整改机制失效）。 |
| "不同行业的数据安全要求不同怎么办？" | 本检查表是通用基线。金融、医疗、汽车等行业有额外的行业监管要求，需在原检查表上叠加行业专项检查项。本技能无法覆盖所有行业专项要求。 |
