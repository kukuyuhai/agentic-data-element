# 🚨 incident-response-playbook

## When to Use

当出现以下任何疑似数据安全事件时，应立即启用本技能启动应急响应：

- 疑似数据泄露（暗网/公开论坛/社交媒体出现内部数据）；
- 未授权访问（非授权账号或 IP 批量访问敏感数据）；
- 异常批量导出（大流量/大文件量异常数据导出）；
- 勒索软件攻击、内部人员违规、第三方安全事件影响我方数据；
- 安全漏洞事件且已有利用迹象。

注意：本技能是**应急处置指南**，不替代正式应急组织体系和法定报告义务。启动后即刻同步通知安全负责人和法务。

## 工作原则

1. **时间第一**：数据泄露应急响应的黄金窗口极短，每个阶段的 SLA 必须严格遵守，不得拖延。
2. **先止损再追责**：第一要务是阻止数据继续外泄，责任追究和根因分析在影响受控后进行。
3. **证据保护**：处置过程中不得破坏可追溯证据（日志、内存快照、磁盘镜像），所有操作需记录留痕。
4. **信息管控**：事件信息仅限应急团队内的"need-to-know"范围传播，避免引发不必要的舆情或打草惊蛇。
5. **依法通报**：达到法定通报门槛的，必须按时限向监管机构、受影响的个人信息主体进行通报。

## 事件严重等级

在响应启动时，首先根据以下标准对事件定级：

| 等级 | 定义 | 典型场景 | 响应 SLA |
|------|------|---------|----------|
| **P0 — 特别重大** | 涉及 100 万+ 条个人信息或核心商业秘密数据泄露；已被公开传播；可能导致严重社会影响 | 核心数据库被拖库、暗网大量出售、媒体已报道 | 即刻启动，全员就位 |
| **P1 — 重大** | 涉及 1万-100万 条个人信息或重要业务数据泄露；已被攻击者获取但未公开 | API 接口被绕过批量下载、内部人员违规外传 | 2小时内启动 |
| **P2 — 一般** | 涉及少量个人信息（1万条以下）或非敏感数据异常；发现漏洞但无利用迹象 | 测试环境数据泄露、个别账号异常访问 | 8小时内启动 |

## 72小时应急响应时间线

### 第0-2小时：识别与定级

**目标**：确认事件真实性、评估影响范围、启动应急组织。

#### 关键动作

1. **事件确认**（15分钟）：核实告警可信度，获取发现时间/涉及数据/异常表现，调取安全日志验证。
2. **影响评估**（30分钟）：确定受影响数据资产范围，判断是否涉及个人信息及是否达到通报门槛（`个人信息保护法（2021）第五十七条`），初步判定攻击路径。
3. **事件定级**（15分钟）：按 P0/P1/P2 定级。P0 通知 CEO/CTO/法务；P1 通知安全负责人和分管副总；P2 通知安全负责人。
4. **应急组织就位**（1小时）：激活应急团队（安全+运维+法务+公关+业务），建立应急沟通渠道。

#### 关键产出物：事件初始报告、应急团队就位确认

### 第2-8小时：遏制

**目标**：阻止数据继续外泄，隔离受影响系统，控制事态扩散。

#### 关键动作

1. **系统隔离**（1小时）：涉事服务器断网隔离（保持开机保留内存证据），冻结涉事数据访问权限，封禁 IP/关闭端口/暂停 API。
2. **止损措施**（2小时）：API 滥用→关闭涉事 API；内部违规→冻结账号保全设备；勒索攻击→断网防横向扩展；第三方泄露→通知暂停合作撤消权限。
3. **证据保全**（持续）：磁盘快照/内存 dump，保存所有安全日志，操作全程留痕。不可直接关机。
4. **初步取证**（并行）：分析攻击路径，提取残留文件（webshell/后门），收集外传数据证据。

#### 关键产出物：隔离确认书、证据包 V1、攻击路径分析（初步）

### 第8-24小时：根除

**目标**：清除攻击残留，修复漏洞，加固防御。

#### 关键动作

1. **恶意组件清除**（4小时）：删除 webshell/后门/持久化机制，恢复系统文件至安全版本，重置所有受影响账号密码和密钥。
2. **漏洞修复**（并行）：修补被利用漏洞，加强缺失的安全控制（网络分段、访问控制）。
3. **大规模排查**（并行）：全量资产同类漏洞扫描，横向排查其他系统是否受影响。
4. **安全加固**（并行）：更新安全策略，收缩访问权限，升级异常行为检测规则。

#### 关键产出物：根除操作记录、漏洞修复确认书、加固措施清单

### 第24-48小时：恢复

**目标**：安全恢复业务运行，持续监控确保无复现。

#### 关键动作

1. **恢复验证**：从安全备份恢复数据，对比差异确认数据完整性，安全扫描确保无残留。
2. **分阶段复运**：内部灰度→内部用户→外部用户，每步观察 2 小时无异常后进入下一步。
3. **强化监控**（此后至少 7 天）：部署专项监控规则，7×24 加强监控。
4. **泄露追踪**：持续监控暗网/社交媒体/通报平台是否有受影响数据被公开。

#### 关键产出物：恢复操作记录、监控加强方案、复运确认书

### 第48-72小时：告知

**目标**：履行法定通报义务，完成对内对外沟通。

#### 关键动作

1. **监管报告**（72小时期内）：按 `个人信息保护法（2021）第五十七条` 要求，报告事件基本情况、已采取措施、个人影响分析、后续整改，报送网信办/行业主管/公安。
2. **用户告知**（并行）：通知受影响个人事件情况、影响、已采取措施和个人可采取的保护行动，选择合适的告知方式。
3. **内部通报**（并行）：向管理层提交总结报告，P0 事件向董事会报告。
4. **对外沟通**（按需）：P0 且已公开传播时，准备官方声明，统一对外口径。

#### 关键产出物：监管报告、用户告知文本、内部总结报告

## 应急组织角色分工

总指挥（CISO/安全 VP）负责总体决策和对外沟通审批；安全分析（安全工程师）负责攻击路径分析和根因定位；系统处置（运维/DevOps）负责隔离、修复、恢复；法律合规（法务/DPO）负责法规判定和监管报告；对外沟通（公关/市场）负责媒体声明和舆情监控；业务协调负责影响评估和客户沟通；记录员负责全程记录时间线与决策。

## 沟通模板

监管报告需包含以下要素（按 `个人信息保护法（2021）第五十七条` 要求）：

**事件基本情况**（类型/时间/涉及数据/初步原因）→ **已采取措施**（遏制/排查/修复）→ **个人影响分析**（受影响人数/可能危害/减轻措施）→ **后续整改计划** → **联系人信息**

监管报告应于确认泄露后 72 小时内提交至网信办或行业主管/公安部门。用户告知需说明事件情况、影响、已采取措施和个人可采取的保护行动。

## Inputs

| 字段 | 类型 | 必填 | 说明 | 示例 |
|------|------|------|------|------|
| `incident_type` | string | 是 | 事件类型 | "数据泄露" / "未授权访问" / "勒索攻击" / "内部违规" |
| `discovery_time` | string | 是 | 发现时间 | "2025-01-15 14:30:00" |
| `discovery_source` | string | 是 | 发现途径 | "安全监控告警" / "外部通报" / "员工报告" / "暗网发现" |
| `affected_systems` | list[object] | 是 | 涉及系统 | `[{系统名称:"用户平台",数据库:"user_db",表:"users,orders"}]` |
| `affected_data_scope` | object | 是 | 受影响数据范围 | `{数据类型:["个人信息","交易记录"],记录数:"约50万条",是否含敏感个人信息:true}` |
| `initial_observation` | string | 是 | 初始发现描述 | "15日14:00起，外部IP 203.0.x.x 通过API批量下载用户表，单小时下载量远超正常值" |
| `current_status` | string | 是 | 当前事态 | "数据仍在持续外传" / "异常已停止但不确定是否已外传" / "已主动阻断" |
| `prior_actions` | list[string] | 否 | 已采取的行动 | `["已通知安全负责人","已暂停涉事API"]` |

## Steps

**前置条件**：用户已报告疑似安全事件并提供了基本的发现信息。

1. **接收并确认事件**：
   - 获取 Inputs 中的所有必填信息
   - 判断是否为误报（false positive），若是则不启动应急流程，记录为安全事件核实记录
2. **事件定级**：
   - 根据受影响数据的数量和敏感度，按 P0/P1/P2 标准定级
   - 输出 `severity_level`
3. **生成应急时间线**：
   - 以发现时间为 T0，生成 72 小时各阶段的时间节点和 SLA
   - 每个阶段列出必做动作清单
4. **分配应急角色**：
   - 按团队结构建议角色分配
5. **输出遏制计划**：
   - 根据事件类型和涉及系统，生成具体的遏制措施清单
6. **输出根除与恢复计划**：
   - 根据攻击路径分析（初步），生成根除和恢复步骤
7. **生成告知草稿**：
   - 如果是 P0 或 P1 事件且涉及个人信息，生成监管报告和用户告知的草稿模板
8. **证据收集指引**：
   - 列出需要保全的证据类型清单
9. **事后整改建议**：
   - 事件结束后，根据根因分析，生成系统性的整改建议和对应技能引用
10. 在整个应急过程中持续追踪时间线，逼近 SLA 时主动提醒。

## 常见问法与处理策略

| 问法 | 处理策略 |
|------|----------|
| "不确定是不是真的泄露了，还要启动应急吗？" | 宁可信其有。根据"疑似泄露"已足够启动识别阶段（0-2小时）。如果在识别阶段确认为误报，直接结束流程并记录。拖延确认可能导致错过遏制黄金窗口。 |
| "P0事件要不要报警？" | 需要。P0事件且涉嫌犯罪的（如黑客攻击、勒索），应立即向公安机关报案。报案不是替代应急响应，而是并行进行的法律程序。报警前务必保全所有电子证据。 |
| "能不能先恢复业务再修复漏洞？" | 绝对不行。在有漏洞未修复的情况下恢复业务等于主动暴露风险，可能导致二次泄露。必须根除完成并安全验证后才能复运。 |
| "已经超过72小时了还没通知监管怎么办？" | 虽然已超时，但仍应立即通知。延迟报告本身可能构成违规，但在通知中应如实说明延迟原因。《个人信息保护法》的72小时是法定期限，超过后应尽快补报并主动说明。 |
| "用户通知要发到什么程度？" | 告知程度应与风险程度匹配。如果风险较小（如仅涉及脱敏后的非敏感信息），可简要告知。如果风险较大（如银行卡信息泄露），应详细说明并建议用户采取换卡等保护措施。注意：告知本身也可能引发恐慌，需措辞谨慎但信息充分。 |
| "应急过程中发现是内部人干的怎么办？" | 先完成技术层面的止损和证据保全，再进行内部追责。涉及员工的，同步通知 HR 和法务，取得法律和劳动关系处理方面的指导。不得私自讨论或传播未经证实的"嫌疑人"信息。 |
