# 🎭 de-identification-toolkit

## When to Use

当出现以下任一数据处理或数据流转场景时，应启用本技能进行去标识化方案设计和评估：

- 数据发布/共享、数据交易前准备、数据分析与建模外包；
- 合规审查、个人信息保护影响评估、数据资产入表前合规确认。

## 工作原则

1. **风险比例原则**：去标识化方法和参数的选择应与数据敏感度和共享场景的风险高低相匹配，不过度也不不足。
2. **效用保留原则**：去标识化在保护隐私的同时需保留数据分析的效用，避免过度泛化导致数据失去分析价值。
3. **方法组合原则**：单一方法存在固有局限，实际场景中通常需要组合使用多种方法（如 k-匿名 + 差分隐私）以达到综合防护效果。
4. **可审计原则**：去标识化的方法和参数选择需有据可查，可在合规审查中追溯论证。
5. **不可逆目标**：最终目标是使数据不能唯一指向特定个人且在合理手段下不可复原（即实现"匿名化"的法律标准 `个人信息保护法（2021）第十五条`）。

## 三类去标识化方法

### 方法一：k-匿名 (k-Anonymity)

#### 原理

将数据集中的每条记录与至少 k-1 条其他记录在准标识符（Quasi-Identifier, QI）上不可区分。通过泛化（Generalization）和抑制（Suppression）两种技术使每个等价类至少包含 k 条记录。

**适用场景**：结构化数据（表格型）、需要保留统计分布特征的分析、数据量较大且准标识符较少的场景。

#### 参数建议

| 数据敏感度 | 推荐 k 值 | 风险控制效果 | 典型场景 |
|-----------|----------|-------------|----------|
| 低敏感（通用行为数据） | k=2~3 | 基本防止直接重识别 | 公开统计报告、人口统计 |
| 中敏感（消费/位置数据） | k=5~10 | 防止链接攻击和重识别 | 商业数据分析、用户画像共享 |
| 高敏感（健康/金融数据） | k≥20 | 高级别防护，大幅降低重识别风险 | 医疗研究、信贷分析、交易数据共享 |

**GB/T 43697-2024 要求**：涉及个人信息的数据共享，k 值应不低于 5。

#### 效用损失评估

- **信息损失度量标准**：分类属性的泛化层级跳数、数值属性的精度损失百分比
- **可接受的效用损失**：通常不超过 15%（即泛化/抑制导致的有效记录或精度损失不超过 15%）
- **效用-隐私平衡**：若 k=5 时信息损失已超过 15%，说明数据稀疏度过高，应考虑降低 k 值并增加其他方法（如差分隐私）作为补偿

#### 实施步骤

1. 识别数据集中所有准标识符（如年龄、性别、邮编等可组合识别个人的属性）
2. 设定目标 k 值
3. 对每个 QI 属性构建泛化层级树
4. 执行 k-匿名算法（Mondrian 算法或 Datafly 算法）
5. 检查结果等价类是否满足 k 要求
6. 计算信息损失
7. 若不满足 k 要求或信息损失过高，调整泛化策略或使用抑制

### 方法二：l-多样性 (l-Diversity)

#### 原理

在 k-匿名基础上增加对敏感属性（Sensitive Attribute）值的多样化要求：每个等价类中的敏感属性至少包含 l 个"表现良好"的不同值。l-多样性主要防止属性链接攻击（即攻击者通过已知准标识符推断出敏感属性值）。

**适用场景**：数据集中包含高风险敏感属性（如疾病诊断、收入水平、宗教信仰），需同时防止重识别和属性链接攻击的场景。

#### 参数建议

| 敏感属性类型 | 推荐 l 值 | 说明 |
|-------------|----------|------|
| 低敏感（消费偏好） | l=2~3 | 基本防止属性推断 |
| 中敏感（地理位置、职业） | l=3~5 | 提供较好的属性多样性 |
| 高敏感（疾病、收入、信用评分） | l≥5 | 严格防止属性链接攻击 |

#### 实施步骤

1. 完成 k-匿名处理
2. 识别数据集中的敏感属性
3. 选择 l-多样性的具体实现形式
4. 检查每个等价类是否满足 l-多样性要求
5. 对不满足的等价类进行进一步泛化或抑制
6. 重新验证 k-匿名性是否仍满足

### 方法三：差分隐私 (Differential Privacy)

#### 原理

通过向查询结果或数据发布物中注入精心校准的随机噪声，确保攻击者无法判断某条特定记录是否在数据集中——无论其记录是否被包含或排除，查询结果的发布概率分布几乎相同。差分隐私用 ε（隐私预算）衡量隐私保护强度：ε 越小，保护越强，但数据效用越低。

**常用机制**：
- **拉普拉斯机制（Laplace Mechanism）**：向数值型查询结果加 Laplace 噪声（噪声尺度 λ=Δf/ε）
- **高斯机制（Gaussian Mechanism）**：向数值型查询结果加 Gaussian 噪声（适用于多次查询）
- **指数机制（Exponential Mechanism）**：用于非数值型查询（如 Top-K 选择）

**适用场景**：频繁查询或动态发布、机器学习模型训练、需要严格数学隐私保证、数据量较大且允许噪声的场景。

#### 参数建议

| 场景 | 推荐 ε | 隐私保证 | 效用影响 |
|------|--------|---------|---------|
| 严格隐私保护（健康、金融公开统计） | ε≤1 | 强隐私保证 | 效用损失较大（10%-20%） |
| 一般隐私保护（商业数据共享） | ε=1~5 | 中等隐私保证 | 效用损失适中（5%-15%） |
| 弱隐私保护（内部数据分析） | ε=5~10 | 基本隐私保证 | 效用损失较小（<5%） |

**行业参考**：Apple ε=4（iOS 数据收集）、Google ε≈1.1（Chrome 遥测）、美国人口普查局 ε≈19.61（全局预算）。

#### 实施步骤

1. 确定查询类型（计数/求和/均值/直方图/机器学习）
2. 计算全局敏感度（Global Sensitivity）Δf
3. 设定隐私预算 ε 和置信参数 δ
4. 选择适当的噪声机制（拉普拉斯/高斯/指数）
5. 生成噪声并注入查询结果
6. 隐私预算消耗追踪（组合定理确保总 ε 不超标）
7. 效用评估：检查噪声是否导致结果不可用

### 方法对比总表

| 维度 | k-匿名 | l-多样性 | 差分隐私 |
|------|--------|---------|---------|
| 防护目标 | 防止重识别 | 防止属性链接 | 防止存在性推断 |
| 隐私保证强度 | 经验性 | 经验性 | 数学可证明 |
| 适用数据类型 | 结构化表格 | 结构化表格（含敏感属性） | 查询/统计/ML |
| 实现复杂度 | 低 | 中 | 高 |
| 效用损失 | 中等（10%-15%） | 中等偏高 | 可控（取决于 ε） |
| 参数调优难度 | 低 | 中 | 高 |
| 多次查询保护 | 否（每次发布均需重新匿名） | 否 | 是（隐私预算可跟踪） |
| 标准依据 | GB/T 43697-2024 | 学术最佳实践 | 学术最佳实践 |
| 适合组合使用 | 是（基础层） | 是（在 k-匿名之上） | 是（叠加层） |

### 方法选择决策树

```
开始
│
├─ 数据是以表格形式共享/发布的吗？
│  ├─ 是 → 继续
│  └─ 否 → 考虑差分隐私（查询接口场景）
│
├─ 是否包含敏感属性（疾病/收入/宗教等）？
│  ├─ 是 → k-匿名（基础）+ l-多样性（强化）
│  └─ 否 → 仅 k-匿名
│
├─ 是否需要动态/多次查询使用数据？
│  ├─ 是 → 叠加差分隐私保护
│  └─ 否 → 已完成
│
├─ 效用损失是否在可接受范围内？
│  ├─ 是 → 完成方案设计
│  └─ 否 → 调整参数或考虑方法替代
│
└─ 输出去标识化方案建议
```

## Inputs

| 字段 | 类型 | 必填 | 说明 | 示例 |
|------|------|------|------|------|
| `data_description` | object | 是 | 数据描述：类型、字段列表、体量、敏感度 | `{类型:"结构化",字段:["年龄","性别","收入","疾病诊断"],体量:"10万条",敏感度:"高"}` |
| `use_scenario` | string | 是 | 使用场景：发布/共享/交易/外包/内部 | "共享" |
| `regulatory_standard` | string | 否 | 遵循的法规标准 | "GB/T 43697-2024" |
| `target_utility` | number | 否 | 可接受的最大效用损失百分比 | 15 |
| `sensitive_fields` | list[string] | 否 | 用户指定的敏感属性字段 | `["疾病诊断","收入"]` |
| `quasi_identifiers` | list[string] | 否 | 用户指定的准标识符字段 | `["年龄","性别","邮编"]` |
| `deploy_constraints` | object | 否 | 部署约束：可用技术栈、性能要求等 | `{language:"Python",max_latency_ms:500}` |

## Outputs

| 字段 | 类型 | 说明 |
|------|------|------|
| `recommended_method` | string | 推荐的主要去标识化方法 | "k-匿名（k=5）+ l-多样性（l=3）" |
| `method_details` | list[object] | 每种方法的详细配置：参数、理由、标准依据 | `[{method:"k-匿名",params:{k:5},rationale:"...",std_ref:"GB/T 43697-2024"}]` |
| `utility_assessment` | object | 效用损失评估：预估信息损失百分比、对数据分析的影响 | `{estimated_loss:12,acceptable:true,analysis_impact:"..."}` |
| `risk_residual` | object | 残余风险评估：去标识化后仍存在的攻击路径和风险概率 | `[{attack:"重识别",probability:"低",mitigation:"k值保障"}]` |
| `implementation_guidance` | object | 实施指引：算法选择、开源工具推荐、代码示例 | `{tools:["ARX","OpenDP"],code_snippet:"..."}` |
| `compliance_checklist` | list[object] | 合规检查清单：是否满足 GB/T 43697-2024 等标准要求 | `[{item:"k值不低于5",status:"满足",ref:"GB/T 43697-2024"}]` |
| `next_actions` | list[object] | 后续建议：PII 扫描、数据分级分类等 | `[{action:"执行PII扫描",skill:"pii-scan"}]` |

## Steps

**前置条件**：用户已提供数据描述和使用场景。

1. **数据理解**：
   - 解析 `data_description`，识别数据类型（结构化/非结构化）
   - 如果用户未提供准标识符和敏感属性，结合领域知识推断
   - 如果用户不确定敏感度，建议先调用 `data-classify` 进行数据分类分级
2. **场景匹配**：
   - 根据 `use_scenario`（发布/共享/交易/外包/内部）确定隐私保护目标等级
   - 识别是否需要满足特定法规标准（如 GB/T 43697-2024）
3. **方法推荐**：
   - 按决策树逻辑选择主要方法
   - 根据数据敏感度推荐具体参数值
   - 标准溯源：GB/T 43697-2024（k-匿名）、个人信息保护法（2021）第十五条（匿名化标准）
4. **组合方案设计**：
   - 判断是否需要方法组合（如 k-匿名 + l-多样性 + 差分隐私）
   - 设计分层防护策略：底层匿名化、中层多样性、顶层差分隐私
5. **效用损失评估**：
   - 估算每种方法的效用损失
   - 检查总效用损失是否在 `target_utility` 范围内
   - 若超出，调整参数并重新评估
6. **残余风险评估**：
   - 识别去标识化后仍可能的攻击路径（重识别、属性推断、存在性推断）
   - 评估残余概率（低/中/高）
   - 给出补充保护建议
7. **实施指引**：
   - 推荐适用的开源工具（ARX、OpenDP、Google DP Library 等）
   - 提供关键参数配置示例
   - 标注部署约束下的工具选择
8. **合规核对**：
   - 对照适用的法规标准逐项核对
9. **输出完整方案**：
   - 按 Outputs 格式整理
   - 如需要 PII 扫描或分类分级，通过 `foundation/handoff-router` 转交至 `pii-scan` 或 `data-classify`

## 常见问法与处理策略

| 问法 | 处理策略 |
|------|----------|
| "去标识化和匿名化有什么区别？" | 去标识化是技术处理过程（去除或替换标识符），处理后数据仍可能通过与其他数据关联而重识别。匿名化是法律标准：经处理使数据无法识别特定个人且不能复原。GB/T 43697-2024 和《个人信息保护法》要求共享数据达到匿名化标准，实践中通过去标识化技术来实现。 |
| "k 值越大越好吗？" | 不是。k 值越大隐私越安全，但信息损失也越大。需在隐私和效用间取得平衡。GB/T 43697-2024 要求 k≥5，更高 k 值适合高敏感场景但可能使数据失去分析价值。 |
| "三种方法能同时用吗？" | 可以且推荐。典型组合：先执行 k-匿名（基础防护），再叠加 l-多样性（属性防护），对查询结果层叠加差分隐私（数学保证）。但方法叠加会增加效用损失和实施复杂度，需权衡。 |
| "去标识化后的数据还算个人信息吗？" | 这取决于去标识化是否达到了"无法识别特定个人且不能复原"的法律标准。如果达到匿名化标准，则不再属于个人信息，不受《个人信息保护法》约束。但实践中这一定性需由法律专家判断。 |
| "怎么样才算'不能复原'？" | 法律上的"不能复原"是功能标准而非绝对标准。需要评估在现有技术条件下、结合所有合理可能利用的信息手段是否能够重新识别。k-匿名和差分隐私等方法是达成这一标准的技术手段。 |
| "差分隐私的 ε 设置多少合适？" | 取决于场景。ε≤1 为强隐私（适合公开统计数据），ε=1~5 为中等（适合商业共享），ε=5~10 为弱隐私（适合内部分析）。Apple 使用 ε=4，可作为商业场景参考。 |
| "去标识化会影响数据资产定价吗？" | 会。去标识化处理程度越高（k 值大、ε 小），数据精度损失越大，数据产品的分析效用可能降低，从而影响定价。需在交易方案设计中纳入去标识化成本和对数据价值的影响。 |
