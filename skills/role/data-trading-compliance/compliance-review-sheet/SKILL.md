# ✅ compliance-review-sheet

合规评审意见书——六维度逐项审查，一票否决制保障数据交易合规底线。

## When to Use

- 数据产品上架前，需要通过合规审查才能挂牌时
- 交易标的涉及个人信息、重要数据、跨境传输等敏感场景时
- 买方/卖方资质需要审核确认时
- 监管要求变化，需要重新评估已上架产品的合规性时

## 工作原则

1. **一票否决**：六个维度中任意一项不通过，整体评审不通过
2. **留痕可审**：每项检查的依据、证据、结论均记录在案，支持事后审计
3. **法规对齐**：审查标准严格对齐《数据安全法》《个人信息保护法》《数据二十条》等法律法规
4. **分级审查**：按数据敏感程度分级（公开/内部/敏感/高敏），敏感度越高审查越严
5. **前置拦截**：合规审查在挂牌前执行，未通过不得挂牌 `§5.2.1`

## 六维度审查

### 一、主体资质审查

**审查目标：** 确认买方/卖方具备合法经营主体资格和行业准入资质。

| 检查项 | 通过标准 | 红线（一票否决） |
|-------|---------|----------------|
| 营业执照 | 在有效期内，经营范围覆盖数据业务 | 无营业执照或已吊销 |
| 法人身份 | 法人身份信息核实通过 | 法人身份造假或涉案 |
| 行业许可证 | 涉及金融/医疗等行业需提供对应许可证 | 无证经营应许可业务 |
| 信用记录 | 无严重失信记录 | 被列入严重失信主体名单 |
| KYC/KYB | 通过 `kyc-kyb-runner` 完成认证 | KYC/KYB 未通过 |

### 二、数据来源审查

**审查目标：** 确认数据来源合法合规，授权链路完整。

| 检查项 | 通过标准 | 红线（一票否决） |
|-------|---------|----------------|
| 来源证明 | 提供数据来源说明及合法获取证明 | 无法证明数据合法来源 |
| 授权链路 | 数据采集已取得用户授权同意（如涉及PII） | 未经授权采集个人信息 |
| 采购合同 | 如为采购数据，需提供采购合同 | 数据来源为非法交易所得 |
| 爬取合规 | 如为爬取数据，需确认不违反robots协议和反不正当竞争法 | 恶意爬取、破坏技术防护措施 |
| 数据所有权 | 确认卖方拥有数据经营权或持有权 | 卖方无权处置该数据 |

### 三、处理链路审查

**审查目标：** 确认数据加工、脱敏、存储过程合规。

| 检查项 | 通过标准 | 红线（一票否决） |
|-------|---------|----------------|
| 加工合规 | 加工过程不违反数据最小化原则 | 过度加工导致隐私泄露 |
| 脱敏到位 | PII 数据经过脱敏/匿名化处理 | 未脱敏的个人信息直接交易 |
| 存储安全 | 存储环境通过等保2.0三级或以上 | 存储环境未通过安全评估 |
| 传输加密 | 数据传输使用 TLS 1.2+ 加密 | 明文传输敏感数据 |
| 处理留痕 | 加工处理全过程留痕可追溯 | 处理过程无记录无法追溯 |

### 四、出境合规审查

**审查目标：** 确认数据交易是否触发数据出境申报义务。

| 检查项 | 通过标准 | 红线（一票否决） |
|-------|---------|----------------|
| 出境判定 | 确认交易是否涉及数据跨境传输 | 涉及出境但未申报 |
| 申报完成 | 如触发出境，已完成网信办申报/备案 | 触发申报义务但未完成 |
| 安全评估 | 通过数据出境安全评估 | 未通过出境安全评估 |
| 标准合同 | 签署数据出境标准合同（如适用） | 未签署标准合同即出境 |
| 认证体系 | 通过个人信息保护认证（如适用） | 涉及PII出境但未认证 |

> 出境判定调用 `cross-border-checker` 执行自动化判定。

### 五、PII 处理审查

**审查目标：** 确认个人信息处理符合《个人信息保护法》要求。

| 检查项 | 通过标准 | 红线（一票否决） |
|-------|---------|----------------|
| PII 扫描 | 通过 `pii-scan` 自动扫描，PII 字段已识别 | 未执行 PII 扫描 |
| 脱敏方式 | 采取去标识化/匿名化措施 | 明文PII直接交易 |
| 单独同意 | 处理敏感PII已取得个人单独同意 | 未经同意处理敏感个人信息 |
| 最小必要 | 交易数据符合最小必要原则 | 交易数据超出业务必要范围 |
| 删除机制 | 约定到期后数据删除机制 | 无数据删除/销毁机制 |

### 六、授权完备审查

**审查目标：** 确认数据产权"三权分置"清晰（持有权/加工使用权/经营权）。

| 检查项 | 通过标准 | 红线（一票否决） |
|-------|---------|----------------|
| 三权归属 | 持有权、加工使用权、经营权归属清晰 | 三权归属存在争议 |
| 权利边界 | 卖方有权交易的具体权利范围明确 | 卖方超出自身权利范围交易 |
| 授权文件 | 授权链路文件完整（从数据源到卖方） | 授权链路断裂 |
| 限制条款 | 数据使用目的、地域、期限限制明确 | 无使用限制条款 |
| 衍生数据 | 衍生数据权属约定清晰 | 衍生数据权属未约定 |

## Inputs

| 输入项 | 类型 | 说明 |
|-------|------|------|
| 产品信息 | Document | 数据产品说明书、字段说明、样本数据 |
| 卖方资质 | Document | 营业执照、行业许可证、KYC/KYB 结果 |
| 数据来源证明 | Document | 采集授权、采购合同、权属声明 |
| 处理链路说明 | Document | 加工流程、脱敏方案、存储环境 |
| PII 扫描报告 | Document | `pii-scan` 输出的 PII 识别报告 |
| 出境判定结果 | Document | `cross-border-checker` 输出的出境判定 |

## Outputs

| 输出项 | 类型 | 说明 |
|-------|------|------|
| 合规评审意见书 | Document | 六维度审查结果 + 综合结论 |
| 整改清单 | List | 附条件通过时的整改事项及期限 |
| 审查证据包 | Archive | 各项检查的依据文件和证据材料 |

## Steps

1. 接收产品上架申请及相关材料
2. 执行主体资质审查（调用 `kyc-kyb-runner` 结果）
3. 执行数据来源审查（核查来源证明和授权链路）
4. 执行处理链路审查（核查脱敏方案和存储安全）
5. 执行出境合规审查（调用 `cross-border-checker` 判定）
6. 执行 PII 处理审查（调用 `pii-scan` 扫描报告）
7. 执行授权完备审查（核查三权分置和授权链路）
8. 汇总六维度结果，输出评审意见书
9. 如有否决项，标注否决原因；如有条件通过，输出整改清单
10. 命中他岗职责时调用 `foundation/handoff-router` 转交

## 常见问法与处理策略

| 常见问法 | 处理策略 |
|---------|---------|
| 合规审查查什么？ | 六维度：主体资质、数据来源、处理链路、出境、PII、授权，任一不过则整体不过 |
| 什么是"一票否决"？ | 六个维度中任意一个触犯红线项，整体评审直接不通过，不可上诉 |
| 附条件通过是什么意思？ | 存在非红线问题但可整改，给予整改期限（通常15-30日），整改后复核通过即可挂牌 |
| 出境审查什么时候适用？ | 买方在境外、或数据服务器在境外、或买方虽在境内但母公司在境外时触发 |
| PII 数据能交易吗？ | 可以，但必须经过脱敏/匿名化处理，且取得用户授权同意，明文PII禁止交易 |
| 三权分置怎么审？ | 核查持有权/加工使用权/经营权归属是否清晰、卖方是否有权交易、授权链路是否完整 |
