# 🚨 breach-response-sop

数据泄露应急 SOP——0-72 小时黄金时间线，依法通报、报送、通知与复盘。

## When to Use

- 平台发生或疑似发生数据泄露、数据被窃取、未授权访问等安全事件时
- 交易数据在交割环节发生丢失、篡改或泄露时
- 卖方或买方报告数据异常使用、数据外泄时
- 监管部门通知或媒体曝光数据安全事件时

## 工作原则

1. **快速响应**：发现事件后立即启动 SOP，0-24 小时内完成初步评估和内部通报
2. **依法报送**：严格遵守《数据安全法》第二十九条和《个人信息保护法》第五十七条的法定时限
3. **证据保全**：第一时间保全日志、数据快照、操作记录，确保证据链完整
4. **用户通知**：涉及个人信息泄露的，依法通知受影响用户
5. **事后复盘**：事件平息后开展复盘，输出整改方案并跟踪落实

## 应急时间线

### 阶段一：0-24h 初步评估与内部通报

**目标：** 快速确认事件性质、影响范围，启动内部应急机制。

| 行动项 | 负责人 | 交付物 | 截止时间 |
|-------|-------|-------|---------|
| 事件发现与上报 | 安全员/值班人员 | 事件初步报告 | 发现后 1h 内 |
| 启动应急小组 | 合规负责人 | 应急小组成立通知 | 上报后 2h 内 |
| 初步影响评估 | 安全工程师 | 影响范围评估报告 | 启动后 4h 内 |
| 数据保全与隔离 | 运维工程师 | 日志快照、系统隔离确认 | 启动后 6h 内 |
| 初步遏制措施 | 安全工程师 | 遏制措施执行记录 | 启动后 12h 内 |
| 内部通报 | 合规负责人 | 内部通报邮件 | 发现后 24h 内 |

**初步遏制措施：**

- 切断泄露源（关闭相关接口、封禁相关账号）
- 隔离受影响系统（下线相关服务、隔离网络）
- 冻结相关交易（暂停涉事产品的交易和交付）
- 保留现场（禁止清理日志、不重启受影响服务器）

### 阶段二：24-48h 深入调查与报送准备

**目标：** 查明泄露路径、数据类型、影响人数，准备监管报送材料。

| 行动项 | 负责人 | 交付物 | 截止时间 |
|-------|-------|-------|---------|
| 深入根因分析 | 安全工程师 | 根因分析报告 | 24h 后 12h 内 |
| 数据影响明细 | 数据工程师 | 受影响数据清单（类型/量级/用户数） | 24h 后 12h 内 |
| 证据链整理 | 合规专员 | 证据保全清单 | 24h 后 18h 内 |
| 监管报送材料准备 | 合规负责人 | 报送报告草稿 | 24h 后 24h 内 |
| 用户通知模板准备 | 合规专员 | 用户通知函草稿 | 24h 后 24h 内 |
| 法律顾问介入 | 法务 | 法律风险评估意见 | 24h 后 12h 内 |

### 阶段三：48-72h 正式报送与用户通知

**目标：** 依法向网信部门报送，通知受影响用户。

| 行动项 | 负责人 | 交付物 | 法定时限 |
|-------|-------|-------|---------|
| 向网信部门报送 | 合规负责人 | 数据安全事件报送报告 | 发现后 72h 内 |
| 通知受影响用户 | 合规专员 | 用户通知函发送记录 | 发现后 72h 内 |
| 通知交易平台 | 合规负责人 | 平台公告（如需） | 报送同步 |
| 通知交易对手方 | 合规专员 | 买方/卖方通知记录 | 报送同步 |
| 配合监管调查 | 合规负责人 | 配合调查记录 | 持续 |

**监管报送报告内容：**

```
数据安全事件报送报告

一、事件概述
   - 发现时间：_______________
   - 事件类型：[泄露/篡改/丢失/未授权访问]
   - 事件等级：[一般/较大/重大]

二、影响范围
   - 受影响数据类型：_______________
   - 受影响数据量：_______________
   - 受影响用户数：_______________
   - 是否涉及个人信息：[是/否]
   - 是否涉及重要数据：[是/否]

三、泄露路径
   - 根因分析：_______________
   - 攻击路径/泄露渠道：_______________

四、已采取的措施
   - 遏制措施：_______________
   - 补救措施：_______________

五、后续计划
   - 整改方案：_______________
   - 预计完成时间：_______________

报送单位：_______________
报送人：_______________
联系电话：_______________
```

**用户通知函模板：**

```
尊敬的用户：

  我们于 YYYY年MM月DD日发现一起数据安全事件，可能涉及您的个人信息。
  现将有关情况通知如下：

  一、事件基本情况
     [事件简述]

  二、可能涉及的信息
     [受影响信息类型清单]

  三、我们已采取的措施
     [遏制和补救措施]

  四、建议您采取的措施
     1. 修改相关账号密码
     2. 关注异常通知和短信
     3. 如有疑问请联系：_______________

  五、我们的联系方式
     服务热线：_______________
     专用邮箱：_______________

  我们对此深表歉意，将全力保障您的信息安全。

  [公司名称]
  YYYY年MM月DD日
```

### 阶段四：72h+ 事后复盘与整改

**目标：** 总结教训，完善防护体系，防止类似事件再次发生。

| 行动项 | 负责人 | 交付物 | 截止时间 |
|-------|-------|-------|---------|
| 事后复盘会议 | 合规负责人 | 复盘会议纪要 | 事件平息后 7 日内 |
| 整改方案制定 | 安全工程师 | 整改方案书 | 复盘后 7 日内 |
| 整改方案执行 | 各责任团队 | 整改完成报告 | 按方案计划 |
| 制度修订 | 合规专员 | 修订后的制度文档 | 整改完成后 15 日内 |
| 赔偿处理（如需） | 法务/财务 | 赔偿方案与执行记录 | 按法律要求 |
| 事件归档 | 合规专员 | 完整事件档案 | 整改完成后 30 日内 |

## Inputs

| 输入项 | 类型 | 说明 |
|-------|------|------|
| 事件初步报告 | Document | 事件发现时间、描述、初步影响 |
| 系统日志 | Archive | 相关系统日志、操作记录 |
| 交易记录 | List | 涉事产品的交易记录 |
| 用户数据 | List | 受影响用户清单（如涉及PII） |

## Outputs

| 输出项 | 类型 | 说明 |
|-------|------|------|
| 事件报送报告 | Document | 向网信部门报送的正式报告 |
| 用户通知函 | Document | 发送给受影响用户的通知 |
| 根因分析报告 | Document | 事件根因、泄露路径、影响范围 |
| 整改方案 | Document | 整改措施、责任人、计划时间 |
| 事件复盘档案 | Archive | 完整事件记录、证据、整改记录 |

## Steps

1. 接到事件报告后立即启动 SOP，成立应急小组
2. 0-24h：初步评估影响范围，执行遏制措施，内部通报
3. 24-48h：深入调查根因，整理证据链，准备报送材料
4. 48-72h：向网信部门正式报送，通知受影响用户
5. 72h+：开展事后复盘，制定并执行整改方案
6. 整改完成后归档事件全记录
7. 命中他岗职责时调用 `foundation/handoff-router` 转交

## 常见问法与处理策略

| 常见问法 | 处理策略 |
|---------|---------|
| 发现数据泄露后多长时间要上报？ | 《数据安全法》要求立即采取补救措施并按规定上报；《个人信息保护法》要求72小时内通知用户和报送网信部门 |
| 什么级别的泄露需要报监管？ | 涉及个人信息、重要数据的安全事件均需报送；一般事件报属地网信办，重大事件报国家网信办 |
| 用户通知必须包含什么？ | 事件基本情况、受影响信息类型、已采取措施、建议用户采取的防护措施、联系方式 |
| 证据保全怎么做？ | 第一时间备份日志、截图、数据快照，存入只读存储，由两人以上签字确认 |
| 事后复盘重点查什么？ | 根因（技术缺陷/管理漏洞/人为失误）、响应时效、处置是否得当、如何预防 |
| 如果是卖方数据泄露怎么办？ | 平台同样启动 SOP，同时通知卖方配合调查，必要时冻结卖方账户和产品 |
