# ✅ deal-9steps-checklist

9 步经纪流程 Checklist——从 intake 到 acceptance 全流程质量卡点，每步定义产出物与通过标准。

## When to Use

- 经纪人推进单笔交易全流程，需要按步骤检查进度和质量时
- 新人经纪人学习交易全流程，需要了解每步做什么和卡点时
- 交易流程出现卡顿，需要定位卡在哪一步时
- 交易审计，需要核查流程合规性时

## 工作原则

1. **步步卡点**：每步设质量卡点，未通过不得进入下一步
2. **产出物明确**：每步必须产出明确的交付物，无产出不算完成
3. **时限可控**：每步设定预估时限，超时触发预警
4. **全程留痕**：每步的操作记录、产出物均归档到 `deal-lifecycle-tracker`
5. **可回溯**：任意时点可回溯交易处于哪一步、谁在处理、卡在哪 `§5.2.5`

## 九步流程详解

### Step 1: Intake（需求采集）

| 项目 | 内容 |
|------|------|
| **目标** | 采集买方结构化需求 |
| **负责角色** | 经纪人 |
| **预估时限** | 1-2 个工作日 |
| **质量卡点** | intake 表单必填字段完整、业务问题描述≥50字、预算合理 |
| **产出物** | `deal-intake-form` 输出的 intake 表单 |
| **通过标准** | 表单完整性检查通过 |
| **调用技能** | `deal-intake-form` |

### Step 2: Matching（供需匹配）

| 项目 | 内容 |
|------|------|
| **目标** | 从挂牌产品中匹配 Top-N 候选卖方 |
| **负责角色** | 经纪人 |
| **预估时限** | 1-2 个工作日 |
| **质量卡点** | 候选产品数 ≥ 3 |
| **产出物** | `matching-scorecard` 输出的打分卡和推荐清单 |
| **通过标准** | 至少 3 个候选产品通过五维度打分且合规非零 |
| **调用技能** | `matching-scorecard` |

### Step 3: Compliance（合规审查）

| 项目 | 内容 |
|------|------|
| **目标** | 买方和卖方通过 KYC/KYB 及合规审查 |
| **负责角色** | 合规专员 |
| **预估时限** | 3-5 个工作日 |
| **质量卡点** | 六维度合规审查均通过或附条件通过 |
| **产出物** | `compliance-review-sheet` 输出的合规评审意见书 |
| **通过标准** | 合规结论为"通过"或"附条件通过"（整改清单已确认） |
| **调用技能** | `compliance-review-sheet`、`kyc-kyb-runner` |

### Step 4: Trial（试用验证）

| 项目 | 内容 |
|------|------|
| **目标** | 买方试用候选产品，验证数据质量和适用性 |
| **负责角色** | 经纪人 + 卖方技术支持 |
| **预估时限** | 7-14 天（按买方需求） |
| **质量卡点** | 试用报告包含数据质量评估和适用性结论 |
| **产出物** | 试用报告（含数据质量评分、字段验证、性能测试） |
| **通过标准** | 买方确认数据满足业务需求，或提出明确调整要求 |
| **调用技能** | — |

### Step 5: Pricing（定价确认）

| 项目 | 内容 |
|------|------|
| **目标** | 确认最终产品、定价方案和合同条款 |
| **负责角色** | 经纪人 |
| **预估时限** | 2-3 个工作日 |
| **质量卡点** | 定价方案在买方预算内，卖方确认接受 |
| **产出物** | 定价方案书（含价格、支付方式、折扣、调整机制） |
| **通过标准** | 买卖双方对定价方案无异议，书面确认 |
| **调用技能** | `price-elasticity-model`（弹性参考） |

### Step 6: Contract（合同签署）

| 项目 | 内容 |
|------|------|
| **目标** | 起草、审核、签署交易合同 |
| **负责角色** | 经纪人 + 法务 |
| **预估时限** | 5-10 个工作日 |
| **质量卡点** | 合同关键条款齐全（标的/权利/对价/保证/救济/争议） |
| **产出物** | 签署完毕的交易合同 |
| **通过标准** | 买卖双方完成签署，合同生效 |
| **调用技能** | `contract-key-clauses`（产权交易时） |

### Step 7: Delivery（交付执行）

| 项目 | 内容 |
|------|------|
| **目标** | 卖方按合同约定交付数据/API/报告 |
| **负责角色** | 卖方 + 经纪人协调 |
| **预估时限** | 按合同约定（通常 3-7 个工作日） |
| **质量卡点** | 交付内容与合同约定一致，交付方式安全合规 |
| **产出物** | 交付确认书（含交付时间、内容清单、方式） |
| **通过标准** | 买方确认收到交付物，交付确认书签署 |
| **调用技能** | — |

### Step 8: Acceptance（验收确认）

| 项目 | 内容 |
|------|------|
| **目标** | 买方验收数据质量、完整性、时效性 |
| **负责角色** | 买方 + 经纪人协调 |
| **预估时限** | 3-5 个工作日 |
| **质量卡点** | 验收报告包含质量评估、完整性检查、适用性确认 |
| **产出物** | 验收报告（含验收结论：通过/附条件通过/不通过） |
| **通过标准** | 验收结论为"通过"或"附条件通过"（整改项已确认） |
| **调用技能** | — |

### Step 9: Maintenance（售后维护）

| 项目 | 内容 |
|------|------|
| **目标** | SLA 生效，进入售后维护期 |
| **负责角色** | 卖方 + 经纪人跟进 |
| **预估时限** | 按合同约定的售后期限 |
| **质量卡点** | SLA 条款明确，售后响应机制建立 |
| **产出物** | SLA 生效确认书、售后响应机制文档 |
| **通过标准** | SLA 生效，买方确认售后通道畅通 |
| **调用技能** | — |

## Inputs

| 输入项 | 类型 | 说明 |
|-------|------|------|
| 买方 intake 表单 | Document | Step 1 的输入 |
| 挂牌产品库 | Database | Step 2 的输入 |
| 买卖方资质 | Document | Step 3 的输入 |
| 试用反馈 | Document | Step 4 的输入 |
| 定价参考 | Document | Step 5 的输入 |

## Outputs

| 输出项 | 类型 | 说明 |
|-------|------|------|
| 9步 Checklist | Document | 完整流程检查清单及状态 |
| 各步产出物 | Archive | 每步的交付物归档 |
| 交易进度报告 | Document | 当前进度和预计完成时间 |

## Steps

1. 交易创建时初始化 9 步 Checklist，分配交易编号
2. 按 Step 1-9 顺序推进，每步完成后更新状态
3. 每步执行质量卡点检查，未通过不进入下一步
4. 每步产出物归档到 `deal-lifecycle-tracker`
5. 超时步骤触发预警，经纪人介入处理
6. 全部 9 步完成后，交易归档 `§5.2.5`
7. 命中他岗职责时调用 `foundation/handoff-router` 转交

## 常见问法与处理策略

| 常见问法 | 处理策略 |
|---------|---------|
| 交易流程分几步？ | 9 步：intake→matching→compliance→trial→pricing→contract→delivery→acceptance→maintenance |
| 某一步卡住了怎么办？ | 查看卡点条件，确认缺失项，联系相关责任人推进；超时 3 天升级处理 |
| 能跳过某些步骤吗？ | 不建议跳过；试用（Step 4）在买方放弃时可跳过，但需书面记录原因 |
| 整个流程多久能完成？ | 正常 20-40 个工作日，紧急交易可压缩到 15 天（需买卖方配合） |
| 质量卡点没过怎么办？ | 不进入下一步，针对卡点问题整改后重新检查，整改超时则交易终止 |
| Checklist 怎么跟踪？ | 录入 `deal-lifecycle-tracker`，每步状态实时更新，支持看板展示 |
