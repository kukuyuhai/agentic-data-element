# 📝 deal-intake-form

需求 intake 表单——买方首次接触时的结构化需求采集，覆盖业务问题、数据需求、预算、时限、授权、合规六维度。

## When to Use

- 买方首次联系平台，需要采集需求信息时
- 买方通过电话/邮件/在线咨询提出数据需求，需要结构化记录时
- 买方需求发生变化，需要更新 intake 表单时
- 经纪人交接客户，新接手经纪人需要了解客户需求全貌时

## 工作原则

1. **结构化采集**：所有需求按固定字段采集，避免自由描述导致信息遗漏
2. **业务问题优先**：先问"要解决什么业务问题"，再问"需要什么数据"，避免买方局限在特定数据产品
3. **预算前置**：在了解需求的同时了解预算范围，避免推荐超出预算的产品
4. **合规同步**：intake 阶段同步采集合规要求（PII/出境/行业监管），为后续合规审查做准备
5. **一表到底**：intake 表单贯穿整个交易流程，后续 matching、pricing、contract 均引用此表 `§5.2.5`

## Intake 表单字段

### 一、买方基本信息

| 字段名 | 类型 | 是否必填 | 说明 |
|-------|------|:-------:|------|
| 企业名称 | Text | 是 | 买方企业全称 |
| 统一社会信用代码 | Text | 是 | 用于 KYB 校验 |
| 联系人姓名 | Text | 是 | 主要联系人 |
| 联系电话 | Phone | 是 | 手机或固话 |
| 联系邮箱 | Email | 是 | 企业邮箱优先 |
| 所属行业 | Enum | 是 | 金融/医疗/交通/零售/制造/政务/其他 |
| 企业规模 | Enum | 是 | 大型/中型/小型/初创 |
| 企业官网 | URL | 否 | 辅助了解买方背景 |

### 二、业务问题描述

| 字段名 | 类型 | 是否必填 | 说明 |
|-------|------|:-------:|------|
| 业务问题 | Text(long) | 是 | 详细描述要解决的业务问题（不少于50字） |
| 预期效果 | Text | 是 | 期望达到的业务效果（如降低成本/提升效率/精准营销） |
| 当前痛点 | Text | 否 | 当前遇到的具体困难和瓶颈 |
| 已有方案 | Text | 否 | 目前已尝试的解决方案及不足 |
| 使用场景 | Text | 是 | 数据将用于什么具体场景 |
| 决策链 | Text | 否 | 买方内部决策链路（谁审批/谁使用/谁付费） |

### 三、数据需求明细

| 字段名 | 类型 | 是否必填 | 说明 |
|-------|------|:-------:|------|
| 数据类型 | Enum | 是 | 企业数据/用户数据/交易数据/行业数据/地理数据/其他 |
| 具体字段 | Text(long) | 是 | 需要的数据字段清单 |
| 记录量级 | Enum | 是 | 万条/十万条/百万条/千万条/亿条+ |
| 时间范围 | DateRange | 是 | 数据覆盖的起止时间 |
| 更新频率 | Enum | 是 | 实时/小时/日/周/月/季度/一次性 |
| 数据格式 | Enum | 是 | CSV/JSON/Parquet/API/报告/其他 |
| 地域范围 | Text | 否 | 数据覆盖的地理范围（全国/某省/某市） |
| 数据质量要求 | Text | 否 | 完整性/准确性/一致性等要求 |

### 四、预算范围

| 字段名 | 类型 | 是否必填 | 说明 |
|-------|------|:-------:|------|
| 预算下限 | Float | 是 | 可接受的最低预算 |
| 预算上限 | Float | 是 | 预算天花板 |
| 货币单位 | Enum | 是 | CNY/USD/其他 |
| 支付方式 | Enum | 是 | 一次性/按月/按季/按年/按调用次数 |
| 预算周期 | Enum | 是 | 年度预算/季度预算/项目预算 |
| 审批状态 | Enum | 否 | 已审批/审批中/未申请 |

### 五、时限要求

| 字段名 | 类型 | 是否必填 | 说明 |
|-------|------|:-------:|------|
| 期望交付日期 | Date | 是 | 买方期望的数据交付时间 |
| 最晚可接受日期 | Date | 是 | 交付时间的底线 |
| 试用期望 | Enum | 否 | 需要试用/不需要试用/待定 |
| 试用期长度 | Enum | 否 | 7天/14天/30天/其他 |
| 紧急程度 | Enum | 是 | 紧急(1周内)/一般(2-4周)/不急(1月+) |

### 六、授权范围

| 字段名 | 类型 | 是否必填 | 说明 |
|-------|------|:-------:|------|
| 使用目的 | Text | 是 | 数据的具体使用目的 |
| 使用期限 | Enum | 是 | 1个月/3个月/6个月/1年/永久 |
| 地域限制 | Enum | 是 | 仅境内/全球/特定区域 |
| 再许可权 | Enum | 是 | 不可再许可/可再许可（需约定） |
| 衍生数据权属 | Enum | 是 | 归买方/归卖方/共有/另行约定 |
| 内部使用人数 | Integer | 否 | 买方内部可使用数据的人数 |

### 七、合规要求

| 字段名 | 类型 | 是否必填 | 说明 |
|-------|------|:-------:|------|
| 是否涉及PII | Enum | 是 | 是/否/不确定 |
| PII处理方式 | Enum | 否 | 脱敏/匿名化/不需处理 |
| 是否涉及出境 | Enum | 是 | 是/否/不确定 |
| 出境目的地 | Text | 否 | 数据出境的目标国家/地区 |
| 行业监管要求 | Text | 否 | 适用的行业监管法规 |
| 合规审查要求 | Enum | 否 | 需要平台合规审查/自行审查/免审 |

## 表单完整性检查

| 检查项 | 标准 | 不通过处理 |
|-------|------|-----------|
| 必填字段完整 | 所有"是"的字段均已填写 | 提示买方补充 |
| 业务问题描述 | 不少于 50 字 | 引导买方详细描述 |
| 预算合理 | 上限 > 下限 > 0 | 确认预算信息 |
| 时限合理 | 最晚日期 > 期望日期 > 今天 | 确认时限信息 |
| 数据需求明确 | 数据类型+字段+量级+格式齐全 | 引导细化需求 |

## Inputs

| 输入项 | 类型 | 说明 |
|-------|------|------|
| 买方口头/书面需求 | Text | 买方通过电话/邮件/在线咨询提出的需求描述 |
| 买方企业信息 | Document | 买方提供的营业执照等企业信息 |
| 买方预算说明 | Text | 买方口头或书面的预算说明 |

## Outputs

| 输出项 | 类型 | 说明 |
|-------|------|------|
| Intake 表单 | Document | 结构化的完整需求表单 |
| 需求摘要 | Document | 一页纸需求摘要，便于内部沟通 |
| 完整性检查结果 | Report | 表单完整性检查报告 |

## Steps

1. 买方首次接触时，经纪人引导买方填写 intake 表单
2. 逐项采集买方信息、业务问题、数据需求、预算、时限、授权、合规
3. 执行表单完整性检查，缺失项提示买方补充
4. 生成需求摘要，便于内部沟通和匹配
5. 将 intake 表单录入 `deal-lifecycle-tracker`，分配需求编号
6. 将表单传递给 `matching-scorecard` 执行供需匹配 `§5.2.5`
7. 命中他岗职责时调用 `foundation/handoff-router` 转交

## 常见问法与处理策略

| 常见问法 | 处理策略 |
|---------|---------|
| intake 表单怎么填？ | 按七维度逐项采集：买方信息→业务问题→数据需求→预算→时限→授权→合规，必填项不可遗漏 |
| 买方不愿意透露预算怎么办？ | 解释预算范围用于精准匹配，可给区间而非精确值；如仍拒绝则按"待确认"记录，matching 时按市场均价推荐 |
| 买方需求不明确怎么办？ | 先问业务问题（要解决什么），再引导到数据需求（需要什么数据），避免买方局限在特定产品 |
| intake 表单能改吗？ | 可以，需求变化时更新表单并重新执行 matching，版本号+1 |
| 表单谁负责填写？ | 经纪人引导买方填写，经纪人负责确认信息完整和准确 |
| intake 和 matching 是什么关系？ | intake 是需求采集，matching 是基于 intake 的供需匹配，intake 是 matching 的输入 |
