# 🗂️ listing-category-designer

挂牌品类设计——产品形态 × 行业领域的品类矩阵与属性体系。

## When to Use

- 交易所新建挂牌目录，需要设计品类分类体系时
- 交易所扩展新行业领域（如新增医疗数据品类），需要定义品类属性时
- 现有品类体系无法满足新产品上架需求，需要重构分类树时
- 运营团队反馈买方搜索效率低，需要优化品类导航时

## 工作原则

1. **形态优先**：先按产品形态（数据集/API/报告/数据服务）划分一级分类，再按行业领域划分二级分类
2. **属性标准化**：同一形态的产品共享属性模板，行业特有属性作为扩展属性
3. **买方视角**：分类体系从买方搜索习惯出发，而非卖方上传习惯
4. **可扩展**：分类树支持动态扩展，新增领域不影响现有分类
5. **最小完备**：每个品类至少包含名称、描述、属性模板、上架要求四要素 `§5.2.1`

## 品类矩阵

| 产品形态 \ 领域 | 金融 | 医疗 | 交通 | 零售 | 制造 | 政务 |
|:-------------:|:----:|:----:|:----:|:----:|:----:|:----:|
| **数据集** | 企业征信数据集 | 脱敏病历数据集 | 路网流量数据集 | 消费画像数据集 | 设备工况数据集 | 人口统计数据集 |
| **API** | 实时汇率API | 药品检索API | 路况查询API | 商品比价API | 设备监控API | 企业信息API |
| **报告** | 行业研究报告 | 药物安全评估报告 | 交通规划报告 | 消费趋势报告 | 供应链分析报告 | 经济运行报告 |
| **数据服务** | 风控建模服务 | 影像分析服务 | 路径优化服务 | 精准营销服务 | 预测性维护服务 | 数据治理服务 |

**数据集属性模板：**

| 属性名 | 类型 | 是否必填 | 说明 |
|-------|------|:-------:|------|
| 数据量级 | Enum | 是 | 万条/十万条/百万条/千万条+ |
| 更新频率 | Enum | 是 | 实时/小时/日/周/月 |
| 数据格式 | Enum | 是 | CSV/JSON/Parquet/Avro |
| 字段数量 | Integer | 是 | 数据集字段数 |
| 时间范围 | DateRange | 是 | 数据覆盖起止时间 |
| 样本数据 | File | 是 | 前100行样本（脱敏） |

## 上架要求

### 通用上架要求（所有品类）

1. 卖方完成 KYC/KYB 认证（调用 `kyc-kyb-runner`）
2. 数据产品通过合规审查（调用 `compliance-review-sheet`）
3. 提供产品说明书（含数据来源、字段说明、使用限制）
4. 提供样本数据或试用接口
5. 明确定价方案

### 行业特有上架要求

| 行业 | 额外要求 |
|------|---------|
| 金融 | 需提供金融业务许可证明；涉及征信数据需征信牌照 |
| 医疗 | 需通过 HIPAA 等效合规审查；PII 数据必须脱敏 |
| 交通 | 涉及地理信息需取得测绘资质；需脱敏处理 |
| 零售 | 消费者画像数据需取得用户授权同意 |
| 制造 | 涉及工业数据需确认无保密协议限制 |
| 政务 | 需取得数据开放授权；敏感数据需脱敏 |

## Inputs

| 输入项 | 类型 | 说明 |
|-------|------|------|
| 交易所定位 | Document | 交易所服务的行业领域范围 |
| 现有品类清单 | List | 已有品类及属性（如有） |
| 买方需求调研 | List | 买方常见搜索关键词、品类偏好 |
| 监管要求 | List | 各行业数据交易的监管约束 |

## Outputs

| 输出项 | 类型 | 说明 |
|-------|------|------|
| 品类分类树 | Document | 多级分类树结构 |
| 属性模板集 | Document | 各产品形态的属性模板 |
| 上架要求清单 | Document | 通用 + 行业特有的上架要求 |
| 品类设计说明 | Document | 设计思路、扩展规则、维护流程 |

## Steps

1. 收集交易所定位、买方需求、监管要求等输入
2. 确定一级分类（产品形态）和二级分类（行业领域）
3. 为每个形态设计属性模板（必填 + 选填）
4. 梳理各行业特有的上架要求
5. 构建完整分类树并验证覆盖度
6. 输出品类设计文档 `§5.2.1`
7. 命中他岗职责时调用 `foundation/handoff-router` 转交

## 常见问法与处理策略

| 常见问法 | 处理策略 |
|---------|---------|
| 挂牌品类怎么分？ | 按产品形态×行业领域矩阵设计，一级形态、二级领域 |
| 新产品该归到哪个品类？ | 先判断产品形态（数据集/API/报告/服务），再判断行业领域，匹配分类树 |
| 属性模板怎么设计？ | 参考各形态属性模板，必填属性覆盖核心信息，选填属性补充行业特征 |
| 不同行业的上架要求一样吗？ | 通用要求一致，各行业有额外要求（如金融需牌照、医疗需脱敏） |
| 品类体系能动态扩展吗？ | 分类树支持动态新增节点，新增品类不影响现有结构 |
