# ⚙️ deal-mechanism-designer

交易机制设计——撮合、协议、招标、拍卖四种机制的价格发现与异常处置。

## When to Use

- 交易所上线新交易品种，需要选择或设计交易机制时
- 现有交易机制成交率低，需要评估是否切换机制时
- 买方/卖方提出特殊交易需求（如大宗交易、竞价采购），需要匹配机制时
- 机制运行中出现异常（如恶意报价、流标），需要完善规则时

## 工作原则

1. **品种适配**：不同数据产品形态适配不同机制（数据集适合协议，API适合撮合，定制服务适合招标）
2. **价格公允**：机制设计应保证价格发现过程的公平、公正、透明
3. **担保交收**：所有机制均接入担保交收（Escrow），买方付款到担保账户后卖方交付
4. **异常可控**：预设异常处置规则（流标/恶意报价/超时未支付等），自动触发处理
5. **合规优先**：机制设计需符合监管要求，异常处置留痕可审计 `§5.2.1`

## 四种交易机制详解

### 一、撮合机制

**适用场景：** 标准化数据产品（如 API 调用、通用数据集），买卖双方多对多。

**价格发现：**

| 规则 | 说明 |
|------|------|
| 价格优先 | 买方出价高的优先成交，卖方报价低的优先成交 |
| 时间优先 | 同价位时，先提交的订单优先成交 |
| 撮合方式 | 连续撮合（实时匹配）或集合撮合（定时批量匹配） |

**担保交收流程：**

1. 买方下单并支付到担保账户
2. 担保账户确认收款后通知卖方交付
3. 卖方交付数据/API开通
4. 买方验收通过后，担保账户释放资金给卖方
5. 超时未验收：7个工作日后自动确认

**异常处置：**

| 异常情况 | 处置规则 |
|---------|---------|
| 买方超时未支付 | 订单自动取消，产品重新挂牌 |
| 卖方超时未交付 | 担保账户退款给买方，卖方信用降级 |
| 买方拒收（无理由） | 扣除 10% 违约金给卖方，余款退还 |
| 系统故障导致错单 | 回滚交易，双方免责 |

### 二、协议机制

**适用场景：** 大宗交易、定制化数据产品、长期合作协议。

**价格发现：** 双方协商定价，平台提供参考价区间（基于历史成交数据）。

**流程：**

1. 买方发起询价，填写需求说明
2. 卖方报价，附带产品说明和交付方案
3. 双方在线协商（平台提供聊天+文档共享）
4. 达成一致后生成协议订单
5. 进入担保交收流程

**异常处置：**

| 异常情况 | 处置规则 |
|---------|---------|
| 协商超时（30天无进展） | 自动关闭询价单 |
| 一方中途退出 | 退出方支付协商保证金（订单金额5%）给对方 |
| 交付物与约定不符 | 买方可拒收，卖方需在15日内整改 |

### 三、招标机制

**适用场景：** 买方发布数据需求，多个卖方竞价投标。

**价格发现：** 买方发布需求预算，卖方密封报价，评标后选定中标方。

**流程：**

1. 买方发布招标公告（需求描述、预算范围、评标标准、截止时间）
2. 卖方提交投标书（方案+报价）
3. 评标委员会按评标标准打分（技术方案40%+价格30%+资质30%）
4. 公布中标结果，生成中标订单
5. 进入担保交收流程

**异常处置：**

| 异常情况 | 处置规则 |
|---------|---------|
| 投标方不足3家 | 流标，买方可重新招标或转协议机制 |
| 中标方放弃 | 没收投标保证金，顺延至第二名 |
| 串标嫌疑 | 暂停招标，启动调查，涉嫌违法移交司法 |

### 四、拍卖机制

**适用场景：** 稀缺/独家数据产品，卖方发布产品，买方竞价。

**价格发现：** 卖方设定起拍价，买方公开竞价（英式拍卖），价高者得。

**流程：**

1. 卖方发布拍卖品（产品说明、起拍价、加价幅度、拍卖时长）
2. 买方报名并缴纳保证金（起拍价的10%）
3. 拍卖开始，买方公开出价
4. 拍卖结束，最高价者中标
5. 中标方支付尾款，进入担保交收流程

**异常处置：**

| 异常情况 | 处置规则 |
|---------|---------|
| 无人报名 | 流拍，卖方可降价后重新拍卖 |
| 中标方超时未付款 | 没收保证金，拍品归第二高价者 |
| 恶意抬价 | 取消违规方资格，保证金冻结 |

## 机制对比表

| 维度 | 撮合 | 协议 | 招标 | 拍卖 |
|------|:----:|:----:|:----:|:----:|
| 价格透明度 | 高 | 低 | 中 | 高 |
| 交易效率 | 高 | 低 | 中 | 中 |
| 适用产品 | 标准化 | 定制化 | 需求明确 | 稀缺独家 |
| 买方/卖方 | 多对多 | 一对一 | 一对多 | 多对一 |
| 定价权 | 市场 | 双方协商 | 买方主导 | 卖方主导 |
| 交易周期 | 分钟级 | 天-周级 | 周-月级 | 天级 |
| 适合规模 | 中小额 | 大额 | 中大额 | 中额 |

## 机制选择指南

```
产品是否标准化？
├── 是 → 买卖双方数量？
│   ├── 多对多 → 撮合机制
│   └── 一对一 → 协议机制
└── 否 → 需求方是谁？
    ├── 买方明确需求 → 招标机制
    └── 卖方独家产品 → 拍卖机制
```

## Inputs

| 输入项 | 类型 | 说明 |
|-------|------|------|
| 产品特征 | Object | 产品形态、标准化程度、稀缺性 |
| 交易规模 | Object | 预估交易金额、频次 |
| 买卖双方画像 | List | 买方/卖方数量、类型 |
| 监管约束 | List | 适用的交易规则限制 |

## Outputs

| 输出项 | 类型 | 说明 |
|-------|------|------|
| 机制设计书 | Document | 选定机制、价格发现规则、担保交收流程 |
| 异常处置规则 | Document | 各类异常的触发条件、处置动作、责任划分 |
| 机制对比评估 | Document | 各机制适配度评估、推荐方案 |
| 系统需求 | Document | 机制实现所需的系统功能与接口 |

## Steps

1. 收集产品特征、交易规模、买卖双方画像等输入
2. 评估四种机制的适配度，输出对比评估表
3. 选定推荐机制，设计价格发现规则
4. 设计担保交收流程（Escrow 机制）
5. 定义异常处置规则矩阵
6. 输出机制设计书 `§5.2.1`
7. 命中他岗职责时调用 `foundation/handoff-router` 转交

## 常见问法与处理策略

| 常见问法 | 处理策略 |
|---------|---------|
| 该用哪种交易机制？ | 按选择指南决策树判断：标准化→撮合/协议，非标→招标/拍卖 |
| 担保交收怎么保证安全？ | 买方付款到平台担保账户，验收通过后才释放给卖方，超时自动处理 |
| 流标了怎么办？ | 招标不足3家流标可重新招标或转协议；拍卖无人报名可降价重拍 |
| 恶意报价怎么处理？ | 系统设定报价偏离阈值（±30%），超限触发人工审核；确认恶意则取消资格 |
| 能不能同时支持多种机制？ | 可以，同一产品可同时挂撮合和拍卖，但需设置互斥规则防止重复成交 |
