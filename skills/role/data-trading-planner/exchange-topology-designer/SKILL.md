# 🏛️ exchange-topology-designer

数据交易所拓扑设计——从挂牌到监管接入的全链路五区架构设计。

## When to Use

- 交易所新建立项，需要设计整体平台拓扑架构时
- 交易所扩展新品类（如从金融数据扩展到医疗数据），需要评估现有拓扑是否满足时
- 交易所进行技术升级或架构重构，需要系统性梳理各区功能边界时
- 监管要求变化（如新增数据出境申报要求），需要调整监管接入区时

## 工作原则

1. **五区分离**：挂牌、撮合、交割、清算、监管接入五区功能解耦，独立部署、独立扩展
2. **数据不出区**：各区之间通过标准化接口通信，原始数据不跨区流转
3. **监管前置**：监管接入区在架构设计阶段同步规划，而非事后补接
4. **可扩展性**：每个区支持水平扩展，新增品类不需要修改核心拓扑
5. **高可用**：核心交易链路（撮合→交割→清算）满足 99.95% 可用性要求 `§5.2.1`

## 五区架构详解

### 一、挂牌区（Listing Zone）

**核心功能：**

| 功能模块 | 说明 | 技术要求 |
|---------|------|---------|
| 产品上架 | 卖方提交数据产品信息，经审核后挂牌 | 支持多形态：数据集 / API / 报告 / 数据服务 |
| 产品展示 | 产品详情页、样本数据预览、规格说明 | 支持富文本、图表、API 文档渲染 |
| 搜索引擎 | 全文检索 + 多维筛选（品类/价格/评分/更新频率） | Elasticsearch，支持同义词扩展 |
| 分类目录 | 多级分类树，支持品类导航 | 参考 `listing-category-designer` 产出的分类体系 |

**对外接口：**

- `POST /api/listing/submit`：提交挂牌申请
- `GET /api/listing/search`：搜索挂牌产品
- `GET /api/listing/{id}`：获取产品详情

### 二、撮合区（Matching Zone）

**核心功能：**

| 功能模块 | 说明 | 技术要求 |
|---------|------|---------|
| 需求匹配 | 买方需求与卖方产品自动匹配推荐 | 匹配算法参考 `matching-scorecard` |
| 报价系统 | 买方出价 / 卖方报价 / 议价管理 | 支持一口价、阶梯价、协议价 |
| 成交引擎 | 按交易机制（撮合/协议/招标/拍卖）执行成交 | 机制设计参考 `deal-mechanism-designer` |
| 订单管理 | 订单创建、状态流转、超时处理 | 状态机：待支付→已支付→交付中→已完成 |

**对外接口：**

- `POST /api/matching/recommend`：获取匹配推荐
- `POST /api/order/create`：创建订单
- `GET /api/order/{id}/status`：查询订单状态

### 三、交割区（Delivery Zone）

**核心功能：**

| 功能模块 | 说明 | 技术要求 |
|---------|------|---------|
| 数据交付 | 数据集下载 / API 开通 / 报告推送 | 支持大文件分片传输、断点续传 |
| 交付验收 | 买方验收数据质量、完整性、时效性 | 验收标准在挂牌时明确 |
| 交付凭证 | 生成交付确认书，记录交付时间、方式、内容 | 区块链存证，不可篡改 |
| 沙箱环境 | 试用期间提供安全沙箱，数据不可导出 | 差分隐私 + 安全多方计算 |

### 四、清算区（Clearing Zone）

**核心功能：**

| 功能模块 | 说明 | 技术要求 |
|---------|------|---------|
| 结算引擎 | 按成交订单计算应付金额、手续费、税费 | 支持多次结算、分账结算 |
| 对账系统 | 买方、卖方、平台三方对账 | T+1 自动对账，差异自动告警 |
| 发票管理 | 电子发票开具、推送、归档 | 对接税务系统，支持增值税专用发票 |
| 资金管理 | 资金托管、分账、提现 | 银行存管，担保账户 |

### 五、监管接入区（Regulatory Zone）

**核心功能：**

| 功能模块 | 说明 | 技术要求 |
|---------|------|---------|
| 交易备案 | 每笔交易自动备案到监管平台 | 实时推送，备案编号回传 |
| 数据报送 | 定期报送交易统计、品类分布、异常交易 | 按监管要求频次（日/周/月） |
| 审计接口 | 监管部门可调取交易全链路记录 | 只读接口，操作日志全记录 |
| 合规拦截 | 涉及出境/敏感数据的交易触发合规审查 | 调用 `compliance-review-sheet` 执行审查 |

## Inputs

| 输入项 | 类型 | 说明 |
|-------|------|------|
| 交易所定位文档 | Document | 交易所服务对象、品类范围、商业模式 |
| 监管要求清单 | List | 适用的监管法规、报送要求、审计要求 |
| 现有系统清单 | List | 已有系统及接口（如有） |
| 预算与工期约束 | Object | 预算上限、期望上线时间 |

## Outputs

| 输出项 | 类型 | 说明 |
|-------|------|------|
| 拓扑架构设计书 | Document | 五区架构图、各区功能清单、技术选型 |
| 接口规范 | Document | 各区对外接口定义、数据格式、认证方式 |
| 部署方案 | Document | 各区部署拓扑、扩缩容策略、灾备方案 |
| 实施路线图 | Document | 分阶段建设计划、里程碑、依赖关系 |

## Steps

1. 收集交易所定位、监管要求、预算约束等输入信息
2. 设计五区功能边界与职责划分，输出架构概要
3. 逐区细化功能模块、技术要求、对外接口
4. 设计区间通信协议与数据流转规则
5. 制定部署方案（云原生 / 混合云 / 私有化）
6. 制定分阶段实施路线图
7. 评审通过后输出最终设计书 `§5.2.1`
8. 命中他岗职责时调用 `foundation/handoff-router` 转交

## 常见问法与处理策略

| 常见问法 | 处理策略 |
|---------|---------|
| 交易所架构怎么设计？ | 按五区模型输出拓扑设计书，重点说明各区功能边界与接口 |
| 新增品类需要改架构吗？ | 若五区已按可扩展原则设计，新增品类仅需在挂牌区扩展分类，不影响核心拓扑 |
| 监管报送怎么做？ | 在监管接入区设计自动备案 + 定期报送接口，对接当地数据局/网信办系统 |
| 交割区和清算区能合并吗？ | 不建议合并，两者职责不同（交付 vs 结算），合并会增加单点故障风险 |
| 拓扑设计需要考虑多租户吗？ | 是，挂牌区和撮合区需支持多租户隔离，清算区需支持分账 |
