# 🛣️ asset-planning-roadmap

## When to Use

当以下场景出现时调用本技能：

- **年度数据资产建设规划**：年初需要制定本年度数据资产建设目标、项目清单和资源分配方案
- **可行性研究立项**：启动数据资产管理平台建设、数据治理专项等大型项目前，需要 3 年路线图作为可研报告核心输入
- **DCMM 贯标升级**：计划从当前 DCMM 等级提升至目标等级时，需要对应路线图规划能力建设节奏
- **数据资产入表准备**：企业计划启动数据资产入表，需要前置的数据建设路线图保障数据可计量、可确权
- **预算周期规划**：向管理层/CFO 申请多年预算时，需要清晰展示分期投入和阶段性成果

## 工作原则

1. **分期递进**：避免"大而全"的一步到位思维，按基础→深化→价值三期递进
2. **依赖清晰**：每个里程碑需标明前置依赖，确保路线图的操作可行性和逻辑自洽
3. **可验收**：每个里程碑必须配有可量化的验收指标，禁止"完成平台建设"等模糊表述
4. **资源务实**：资源需求按实际市场价格估算，避免拍脑袋和"先报大数再说"
5. **标准对齐**：所有能力建设对标 `GB/T 36073-2018` 八大能力域和 `T/MIITEC 025-2024 §5.1.2`

## 三年路线图框架

### Year 1：基建期 (Foundation)

**目标**：建立数据资产管理的基础设施和标准体系，达到"数据可见、可管"。

| 里程碑 | 前置依赖 | 验收指标 (KPI) | 资源需求 |
|-------|---------|---------------|---------|
| M1.1 数据资产目录上线 | 数据源梳理完成 | 核心业务系统覆盖率 ≥ 80%；目录搜索响应 < 2s | 平台 License + 2 实施顾问（6 个月） |
| M1.2 数据标准发布 V1.0 | 业务域划分完成 | 标准覆盖率 ≥ 70%（核心指标的命名、口径统一） | 1 数据治理经理 + 3 业务域代表（4 个月） |
| M1.3 数据分类分级完成 | 数据盘点完成 | 全域标记率 ≥ 95%（安全等级 L1-L4 标记） | 1 合规顾问 + 1 安全工程师（3 个月） |
| M1.4 基础平台搭建 | 架构方案审批 | 平台高可用 ≥ 99.5%；数据接入通道 ≥ 50 条 | 3 数据工程师 + 平台实施费（12 个月） |
| M1.5 数据管理组织建立 | 高层审批 | 覆盖 ≥ 3 个核心业务域，数据 Owner 任命 100% | 1 CDO + 数据治理委员会成立 |

**核心投入**：人力 6-8 人，工具平台 License 与实施费，合规咨询费。
**关键风险**：业务域参与度不足 → 需获得 CXO 级别的推进承诺；平台选型失误 → 建议 POC 验证后再采购。

### Year 2：深化期 (Deepening)

**目标**：从"可管"到"管好"，深化数据质量和血缘治理，启动数据共享和应用。

| 里程碑 | 前置依赖 | 验收指标 (KPI) | 资源需求 |
|-------|---------|---------------|---------|
| M2.1 数据质量监控体系上线 | M1.1 目录上线 | 质量规则覆盖率 ≥ 90%（覆盖 6 性检查）；自动化告警率 ≥ 80% | 1 数据质量工程师 + 规则引擎平台（6 个月） |
| M2.2 数据血缘图谱 V1.0 | M1.1 目录 + M2.1 质量 | 核心链路血缘覆盖率 ≥ 95%；影响分析时间 < 30s | 血缘工具 License + 1 元数据工程师（4 个月） |
| M2.3 数据共享服务门户上线 | M1.1 目录 + M1.3 分级 | 共享申请 T+1 处理率 ≥ 80%；API 接口数 ≥ 20 | 1 API 开发 + 1 产品经理（6 个月） |
| M2.4 DCMM 三级达标 | M1.1-M1.5 所有里程碑 | 8 个能力域评分均 ≥ 3 级 | 1 咨询顾问（4 个月贯标辅导） |
| M2.5 首批数据产品交付 | M2.3 共享门户 | ≥ 3 个数据产品上线（如客户画像、风控评分等） | 2 数据分析师 + 1 产品经理（持续） |

**核心投入**：人力 8-10 人（含新增的数据产品方向），平台扩展及运维费。
**关键风险**：数据产品缺乏业务需求方 → 需并行建立业务需求对接机制；DCMM 贯标延期 → 预留 3 个月缓冲。

### Year 3：价值期 (Value Realization)

**目标**：从"管好"到"变现"，实现数据资产交易、入表和估值。

| 里程碑 | 前置依赖 | 验收指标 (KPI) | 资源需求 |
|-------|---------|---------------|---------|
| M3.1 数据资产入表完成 | M1.3 分级 + M2.4 DCMM 三级 + M2.5 产品 | 首批 ≥ 5 项数据资产入表；入表金额 ≥ ¥xxx 万 | 1 会计 + 1 评估师 + 审计配合 |
| M3.2 数据交易接口上线 | M2.3 共享门户 + M3.1 入表 | 对接 ≥ 2 个数据交易所；首年交易额 ≥ ¥xxx 万 | 1 交易运营 + 合规审批 |
| M3.3 数据资产估值体系建立 | M3.1 入表 | 完成 ≥ 3 项核心资产的估值（成本法/收益法/市场法） | 可调用 `role/data-asset-appraiser` 系列技能 |
| M3.4 DCMM 四级 / 五级冲刺 | M2.4 三级达标 | 8 个能力域评分均 ≥ 4 级 | 1 咨询顾问（6 个月） |
| M3.5 数据资产运营常态化 | M2.5 产品 + M3.1 入表 | 季度数据资产报告常态化出具；数据收入同比增长 ≥ 30% | 1 数据运营经理（持续） |

**核心投入**：人力 10-12 人（含交易运营和会计角色），评估审计费，交易所接入费。
**关键风险**：入表合规审计不通过 → 前置审计预审机制；交易量不达预期 → 多元化产品线和销售渠道。

|----|----|    |----|----|    |----|----|

M1.1 [====]基

M1.3 [==]基
M1.4 [============]基
M1.5 [===]基
            M2.1 [=====]深
            M2.2 [===]深
            M2.3 [=====]深
            M2.4    [===]深
            M2.5       [=====]深

## Inputs

| `priority_domains` | array | 否 | 优先推进的能力域（DCMM 八大域中选） | `["数据战略", "数据治理", "数据质量"]` |

## Steps

### Step 1：基线评估

1. 获取当前 DCMM 能力域评分（可调用 `management/dcmm-scoring`）
2. 确认目标 DCMM 等级和达成时间
3. 识别现有平台和工具清单，评估重用和补充需求
4. 确认组织规模和行业特征，据此校准资源估算

### Step 2：能力差距分析

1. 对比当前态和目标态的能力差距（可调用 `role/data-asset-planner/capability-gap-analysis`）
2. 识别关键短板域（差距最大的 2-3 个能力域）
3. 将差距转化为改进主题和优先行动

### Step 3：三期里程碑设计

1. 将改进行动分配到三期（基建/深化/价值）
2. 为每个里程碑设计前置依赖检查
3. 为每个里程碑设定可量化的验收 KPI
4. 估算每个里程碑的资源需求（人力、工具、咨询）

### Step 4：依赖关系与路径校验

1. 绘制里程碑依赖关系图，确保逻辑自洽
2. 检查是否存在循环依赖或无法满足的前置条件
3. 标注关键路径（耗时最长的依赖链）
4. 预留每期 2-3 个月的缓冲时间

### Step 5：甘特图与资源汇总

1. 生成时序视图（甘特图，标注三期边界）
2. 汇总分年度资源需求（人力、工具、咨询费用）
3. 与可用预算校准，超出部分提出优先级裁剪建议

### Step 6：风险与建议输出

1. 识别每期的关键执行风险
2. 制定缓解措施和预案
3. 输出最终路线图文档和汇报材料

## 常见问法与处理策略

| 用户问法 | 处理策略 |
|---------|---------|
| "帮我们制定 3 年数据建设路线图" | 先问现状（DCMM 等级、平台情况、预算），再做差距分析，然后按三期框架生成路线图 |
| "为什么一定要 3 年？我们想一年搞定" | 解释数据资产建设的递进规律：基建→深化→价值不可跳跃，建议把 3 年中各期压缩但保留依赖关系 |
| "里程碑的验收指标怎么定？" | 每个里程碑提供 2-3 个验收 KPI 示例，引导用户根据企业实际情况调整目标值，可对标同行业企业 |
| "资源需求怎么估算？" | 按里程碑逐项估算人力（岗×周期）、工具（市场价参考）、咨询（按人天报价）。建议预留 15% 弹性空间 |
| "路线图更新频率是多久？" | 建议每季度回顾和滚动更新，保持里程碑不变但可调整资源分配和时序 |
| "如果第一阶段建设滞后怎么办？" | 评估对后续里程碑的级联影响，标出被阻塞的依赖项，给出并行推进或裁剪方案 |
| "怎么和业务方对齐路线图？" | 将里程碑翻译为业务方能理解的语言（如"M1.1 数据目录 = 你能在系统里搜到所有你需要的数据"），组织跨部门对齐会 |
