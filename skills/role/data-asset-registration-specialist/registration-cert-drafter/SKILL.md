# 📜 registration-cert-drafter

## When to Use

当需要起草数据资产登记相关的正式文件时启用本技能：

- 准备向地方数据交易所（如北京国际大数据交易所、上海数据交易所、深圳数据交易所）提交登记申请；
- 准备向知识产权中心申请数据知识产权登记；
- 企业内部数据资产入表前的权属登记建档；
- 配合证据链（`evidence-chain-builder` 输出）提供登记前所需的标准格式文件；
- 数据资产产权发生变更时需要重新出具登记证书草稿。

注意：本技能只生成登记申请材料的**草稿**，最终有效版本需由登记机构审核确认。不同登记机构可能有不同的格式要求，草稿需按目标机构的模板进行调整。

## 工作原则

1. **标准化格式**：草稿严格按照登记机构的通用要素编制，减少格式驳回风险。
2. **信息一致性**：草稿中引用的权属信息必须与 `three-rights-analyzer` 和 `evidence-chain-builder` 的输出一致，不得出现矛盾。
3. **可审核性**：每个声明项均需能追溯到证据材料，登记审查人员可以逐一核验。
4. **留有余地**：权利范围描述应准确而不夸大；不确定的部分如实标注"待补充"或"依合同约定"。
5. **针对性适配**：根据目标登记机构的特点调整草稿内容（如不同交易所对数据产品描述的要求不同）。

## 登记证书结构

### 第一模块：登记主体信息

本模块载明登记申请人（数据资产权利人）的基本信息。

| 字段 | 说明 | 示例 |
|------|------|------|
| 主体名称 | 企业全称或个人信息主体名称（与营业执照/身份证一致） | "XX科技有限公司" |
| 统一社会信用代码 | 企业的统一社会信用代码（18位） | "91110000XXXXXXXXXX" |
| 法定代表人/负责人 | 法定代表人或主要负责人姓名 | "张三" |
| 主体类型 | 数商/数据需求方/数据供给方/个人 | "数商" |
| 注册地址 | 营业执照注册地址 | "北京市海淀区XX路XX号" |
| 联系人及方式 | 负责登记事宜的联系人姓名、电话、邮箱 | "李四，13800138000" |
| 数据安全合规资质 | 等级保护、ISO 27001、DSMM 等相关资质 | "等保三级（编号XXX）" |

### 第二模块：登记客体信息

本模块载明被登记的数据资产本身。

| 字段 | 说明 | 示例 |
|------|------|------|
| 数据资产名称 | 登记的数据资产正式名称 | "电商用户消费行为画像数据集" |
| 数据资产描述 | 数据内容概要、用途场景 | "包含XX平台2024年度500万用户浏览、点击、购买行为数据经脱敏和聚合分析后的用户画像" |
| 数据资产标识 | 企业内部资产编码或登记机构分配码 | "DA-2024-E-0001" |
| 数据类型 | 结构化/半结构化/非结构化 | "结构化" |
| 数据体量 | 记录数/存储量 | "约500万条，存储量约200GB" |
| 更新频率 | 静态/实时/周期性 | "每日增量更新" |
| 数据来源 | 自采集/外购/公开/AI生成/公共数据 | "自采集+部分外购补充" |
| 分类分级 | 数据安全分类分级结果 | "商业秘密级（内部管理级）" |
| 技术描述 | 数据格式、Schema、API 规范等 | "CSV格式，包含32个字段，详见数据字典" |
| 去标识化说明 | 是否经过去标识化处理及所采用的方法 | "已通过 k-匿名（k=5）和差分隐私（ε=1）处理" |

### 第三模块：权能范围

本模块是登记证书的核心，直接引用 `three-rights-analyzer` 的分析结论。

| 字段 | 说明 |
|------|------|
| 数据资源持有权 | 持有主体、持有方式（独有/共有）、持有合法性依据 |
| 数据加工使用权 | 加工主体、加工方式、是否可再许可、合同依据 |
| 数据产品经营权 | 经营主体、经营方式（自营/许可/合作）、经营地域和期限限制 |
| 权属限制条件 | 是否存在第三方权利主张、担保权益、查封冻结等限制 |

**权利矩阵示例**：

| 权利类型 | 权利人 | 状态 | 限制 |
|----------|--------|------|------|
| 持有权 | A公司 | 确定 | 无 |
| 加工使用权 | B公司（授权） | 确定 | 仅限境内加工，截止2026-12-31 |
| 经营权 | A公司 | 确定 | 禁止向竞品公司转授权 |

### 第四模块：登记期限

| 字段 | 说明 | 示例 |
|------|------|------|
| 登记生效日期 | 申请登记的起始日期 | "2025-01-01" |
| 登记有效期 | 登记的有效期间 | "3年" |
| 登记截止日期 | 有效期届满日 | "2027-12-31" |
| 续期方式 | 到期后是否可续期、续期条件 | "到期前30日申请续期，续期需重新提供更新后的证据材料" |

### 第五模块：附属证据清单

本模块列出支撑登记的完整证据材料目录，直接引用 `evidence-chain-builder` 的输出。

| 证据编号 | 证据名称 | 证据类型 | 对应权能 | 核验方式 |
|----------|----------|----------|----------|----------|
| E-001 | 数据采购合同 | 来源合法性 | 持有权 | 原件核验 |
| E-002 | ETL 日志（2024年度） | 加工留痕 | 加工使用权 | 系统导出+区块链存证 |
| E-003 | 三权分置声明书 | 权属声明 | 全部 | 原件核验 |
| ... | ... | ... | ... | ... |

### 第六模块：声明确认与签章

本模块为登记申请人的法律声明：

- **真实性声明**：申请人保证所提交的全部材料真实、准确、完整，无虚假记载、误导性陈述或重大遗漏；
- **合法性承诺**：承诺数据来源合法，不侵犯他人合法权益，不违反法律法规强制性规定；
- **持续披露义务**：承诺在登记有效期内，如权属状态发生重大变化（转让、许可、质押、争议等），及时向登记机构报告更新；
- **签章栏**：申请人/法定代表人签字并加盖公章，注明日期。

## Inputs

| 字段 | 类型 | 必填 | 说明 | 示例 |
|------|------|------|------|------|
| `three_rights_result` | object | 是 | `three-rights-analyzer` 的输出结果 | `{holding_right:{...},processing_right:{...},operation_right:{...}}` |
| `evidence_catalog` | list[object] | 是 | `evidence-chain-builder` 输出的证据材料目录 | `[{证据编号:"E-001",证据名称:"...",...}]` |
| `applicant_info` | object | 是 | 登记申请人基本信息 | `{主体名称:"XX科技有限公司",统一社会信用代码:"...",...}` |
| `data_asset_info` | object | 是 | 数据资产基本信息 | `{名称:"...",描述:"...",类型:"结构化",体量:"500万条",...}` |
| `target_exchange` | string | 是 | 目标登记机构 | "北京国际大数据交易所" / "上海数据交易所" / "国家知识产权局" |
| `registration_period` | object | 否 | 登记期限信息 | `{生效日期:"2025-01-01",有效期:"3年"}` |
| `special_requirements` | list[string] | 否 | 目标交易所的特殊格式要求 | `["需提供数据安全合规自评报告","需附区块链存证编号"]` |

## Outputs

| 字段 | 类型 | 说明 |
|------|------|------|
| `certificate_draft` | object | 完整的登记证书草稿（六大模块齐全） |
| `application_form_draft` | object | 登记申请表填写草稿（如目标机构有固定表格） |
| `material_checklist` | list[object] | 申请材料准备清单，标注每项材料的完成状态 |
| `review_notes` | list[string] | 审核预检建议（标注可能被驳回的风险点） |
| `target_exchange_instructions` | string | 目标机构的提交指引（提交方式、流程、费用等） |
| `next_actions` | list[object] | 后续行动建议 |

## Steps

**前置条件**：用户已获得 `three-rights-analyzer` 和 `evidence-chain-builder` 的输出结果。

1. **确认输入完整性**：检查 `three_rights_result` 和 `evidence_catalog` 是否齐全，若不齐则引导用户先完成前置分析。
2. **选择目标机构**：确认目标登记机构，查对应的格式要求和提交规范。
3. **编制第一模块——登记主体信息**：
   - 填写申请人的完整信息
   - 核验统一社会信用代码等关键字段准确性
4. **编制第二模块——登记客体信息**：
   - 填写数据资产描述
   - 若涉及去标识化，引用 `de-identification-toolkit` 的方法说明
5. **编制第三模块——权能范围**：
   - 从 `three_rights_result` 中提取三权归属
   - 生成权利矩阵
   - 标注限制条件和风险提示
6. **编制第四模块——登记期限**：
   - 如用户未提供，按目标机构惯例建议默认期限（通常1-3年）
7. **编制第五模块——附属证据清单**：
   - 从 `evidence_catalog` 中提取证据编号和描述
   - 按三大层级分组排列
8. **编制第六模块——声明确认与签章**：
   - 生成标准声明文本
9. **生成审核预检建议**：
   - 检查草稿中是否存在信息不一致
   - 检查是否有必须字段缺失
   - 标注可能被登记机构驳回的风险点
10. **输出完整草稿**：将六大模块合并输出，附目标机构提交指引。

## 常见问法与处理策略

| 问法 | 处理策略 |
|------|----------|
| "不同交易所的格式不一样怎么办？" | 本技能生成的是通用要素草稿。提交前需根据目标交易所的具体模板格式进行调整。指定 `target_exchange` 参数后，技能会尽量适配该交易所的已知要求。 |
| "登记被驳回了怎么改？" | 查看驳回通知书中的具体原因。常见驳回原因：权属描述不清（→ 返回 `three-rights-analyzer` 重新分析）、证据链不完整（→ 返回 `evidence-chain-builder` 补证）、主体信息与工商登记不一致（→ 修正并重新提交）。 |
| "数据产品还没开发完可以登记吗？" | 可以登记数据资源（如原始数据集或中间加工结果），但数据产品需要具备产品化形态后才能登记为数据产品。建议先登记数据资源，待产品上线后补充登记数据产品。 |
| "登记有效期到了数据还在用怎么办？" | 在有效期截止前向登记机构提交续期申请。续期通常需要提供最新的权属状态证明和持续使用的证据。本技能可生成续期申请草稿。 |
| "持有权共有时登记证书怎么写？" | 在权能范围模块中明确列出所有共有人的名称和共有比例（如有约定）。共有人中的任一一方均可作为申请人，但需取得其他共有人的书面同意。 |
| "数据资产发生权属变更了，证书要重新登记吗？" | 是的。数据产权转让、继承、企业合并分立等导致权属变更的，需要办理变更登记或注销原登记后重新登记。本技能可协助生成变更登记申请草稿。 |
