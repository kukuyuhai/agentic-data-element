# 🔗 evidence-chain-builder

## When to Use

当出现以下任一情形时应启用本技能构建证据链：

- 申请数据资产登记：登记机构要求提交权属证明材料；
- 发生数据权属争议：需要系统性举证证明权利来源；
- 数据资产入表前审计：审计师要求追溯数据资产的合法来源和加工过程；
- 数据交易/授权前尽职调查：交易对手方需要审查权属证据的完整性和可信度；
- 数据合规审查：监管机构要求提供数据全生命周期合规证据；
- 已完成三权分析（`three-rights-analyzer`），需要将分析结论转化为可提交的证据材料清单。

## 工作原则

1. **三层递进**：证据链按"来源合法 → 加工有痕 → 权属声明"三层递进，缺一不可。
2. **原件优先**：优先收集可核验的原件或电子存证，复印件/扫描件需附可核验方式。
3. **时间连续**：证据之间在时间线上必须形成无缝衔接，不能有"证据真空期"。
4. **交叉验证**：同一事实应尽量取得两份以上独立来源的证据互相印证。
5. **可追溯**：每份证据均标注来源、生成时间、保管人和核验途径。
6. **缺口驱动**：发现证据缺失时不直接判定失败，而是列出具体缺口和补证建议。

## 证据链三层体系

### 第一层：来源合法性证据

本层证明数据获取的起点合法，是整个证据链的根基。

#### 证据类型与收集指引

| 子类 | 典型证据 | 关键审查点 | 收集优先级 |
|------|----------|-----------|-----------|
| 采购/交易合同 | 数据采购合同、数据服务协议、数据许可协议 | 合同签署主体与权属主体一致；合同明确约定数据权利归属和使用范围 | P0（必须） |
| 用户授权 | 用户知情同意书、隐私政策接受记录、授权记录日志 | 采集目的与授权范围一致；用户可撤回同意的机制存在 | P0（必须） |
| 采集合规证明 | 个人信息保护影响评估报告、算法备案凭证、行业许可/牌照 | 采集行为符合所涉行业的监管要求 | P1（强烈建议） |
| 公开数据来源说明 | 公开数据集的元信息说明、爬虫协议遵守证明、来源网站的使用条款 | 公开数据的爬取/收集方式符合 robots.txt 和网站条款 | P1（公开数据场景必须） |
| 政府公共数据授权 | 公共数据授权运营协议、开放数据使用许可证 | 授权范围与使用方式一致 | P0（公共数据场景必须） |

#### 来源合法性的红线（一票否决）

以下情形直接判定来源不合法，证据链中断：
- 数据来源于黑客攻击、内网渗透等非法手段
- 未经用户知情同意采集敏感个人信息（《个人信息保护法》）
- 未经授权爬取明确声明禁止爬取的网站数据
- 购买明知来源于非法途径的数据

### 第二层：加工留痕证据

本层证明数据在持有期间经历了明确的加工处理过程，形成可审计的处理日志。

#### 证据类型与收集指引

| 子类 | 典型证据 | 关键审查点 | 收集优先级 |
|------|----------|-----------|-----------|
| ETL 日志 | ETL 任务执行日志、调度记录、报错日志 | 日志覆盖每次加工操作；日志包含操作人、时间、输入输出数据源 | P0（必须） |
| 数据血缘 | 字段级/表级血缘关系图、血统元数据 | 血缘链路覆盖从原始数据到最终产物的全程；血缘与 ETL 日志交叉一致 | P1（强烈建议） |
| 处理记录 | 数据处理 SOP、版本变更记录、数据质量报告 | 处理步骤有文档化记录；版本号可追溯 | P1（强烈建议） |
| 加工资源证明 | 计算资源使用记录、算法模型版本、标注人员记录 | 加工投入可量化证明 | P2（辅助） |

#### 血缘图的生成

建议调用 `management/lineage-graph` 技能生成可视化血缘图，要求：
- 展示字段级别转换关系
- 标注每个加工节点的操作主体和时间
- 区分手工加工与自动化加工
- 标注外部数据引入点

### 第三层：权属声明证据

本层是对前两层证据的整合和权属结论的法律性声明。

#### 证据类型与收集指引

| 子类 | 典型证据 | 关键审查点 | 收集优先级 |
|------|----------|-----------|-----------|
| 登记证书 | 数据资产登记证书、数据知识产权登记证书 | 登记信息与实际情况一致；证书在有效期内 | P0（建议获取） |
| 权属协议 | 三权分置声明书、共有权属协议、收益分配协议 | 协议已明确三权的归属和权利范围；协议经各方签署 | P0（必须） |
| 合同条款 | 委托加工合同中的权属条款、许可合同中的授权条款 | 合同条款清晰明确；不存在歧义或与事实矛盾 | P0（必须） |
| 司法/仲裁裁决 | 法院判决书、仲裁裁决书 | 裁决已生效；裁决内容与主张的权属一致 | P0（如有） |

#### 三权分置声明书核心要素

基于 `three-rights-analyzer` 的分析结果，起草三权分置声明：
1. **持有权声明**：申明持有主体、持有方式（独有/共有）、数据存储位置、安全管理措施
2. **加工使用权声明**：申明加工主体、加工方式、加工成果的知识产权归属
3. **经营权声明**：申明经营主体、经营方式（自营/授权/合作）、经营产品的范围和限制
4. **权属承诺**：各主体承诺所声明的内容真实、完整、无争议
5. **签署页**：各主体加盖公章，注明日期

## 证据完整性检查清单

使用此检查清单逐项核查证据链的完整性：

### 来源合法性维度
- [ ] 数据采购/采集合同是否已归档并可查阅？
- [ ] 个人信息的用户授权/知情同意是否可追溯并满足最小必要原则？
- [ ] 如为公开数据，是否有来源说明和采集合规性证明？
- [ ] 如为公共数据，是否有授权运营协议？
- [ ] 来源合同中的权利条款是否明确约定数据权利归属？

### 加工留痕维度
- [ ] ETL 日志是否覆盖从数据入库到当前状态的全程？
- [ ] 血缘图是否能展示字段级转换关系？
- [ ] 每次重大加工变更是否有版本记录？
- [ ] 加工操作是否与加工使用权的主体授权一致？

### 权属声明维度
- [ ] 是否有明确的三权分置声明书或权属协议？
- [ ] 登记证书（如有）上的信息是否与权属协议一致？
- [ ] 合同中的权属条款是否与三权分析结论一致？
- [ ] 是否有第三方对该数据资产提出过权利主张？

### 证据形式维度
- [ ] 所有关键证据是否均为原件或可核验副本？
- [ ] 电子证据是否有时间戳或区块链存证？
- [ ] 合同等法律文件是否已完成签署盖章？
- [ ] 证据的时间线是否连续、无真空期？

## 缺口识别与补证建议

当检查清单中出现"否"项时，按以下策略处理：

| 缺口类型 | 严重程度 | 对登记的影响 | 补证建议 |
|----------|---------|-------------|----------|
| 来源合法性证据缺失 | 严重 | 登记申请可能被驳回 | 补签补充协议；追溯获取授权；委托律师做法律论证 |
| ETL 日志不完整 | 中等 | 登记审核可能要求补充 | 从系统日志中回溯补充；引入数据血缘工具 |
| 权属协议未签署 | 严重 | 无法完成登记 | 组织各方签署；如需法律意见，建议外部律师介入 |
| 登记证书过期 | 轻微 | 不影响新登记，但失去既有登记效力 | 向原登记机构申请延续 |
| 血缘图缺失 | 轻微 | 不影响登记但影响评估专业性 | 调用 `management/lineage-graph` 生成 |

## Steps

**前置条件**：用户已提供数据资产标识、名称和已有证据清单。

1. **接收三层分析输入**：如果提供了 `three_rights_result`，则以其为证据链构建的起点；否则，先引导用户完成三权分析。
2. **第一层构建——来源合法性证据整理**：
   - 识别数据的来源类型（采购、授权、采集、公开、公共数据）
   - 对照来源类型，逐一收集对应的证据
   - 审查关键条款：权属约定、授权范围、合规证明有效期
   - 标注溯源：`T/MIITEC 025-2024 §5.1.3`
3. **第二层构建——加工留痕证据整理**：
   - 收集 ETL 日志、处理记录等加工证据
   - 若用户需要血缘图，引导调用 `management/lineage-graph`
   - 核验加工痕迹与加工使用权声明的一致性
4. **第三层构建——权属声明证据整理**：
   - 收集已有的权属协议、登记证书
   - 若缺失三权分置声明，提示需要起草（可调用 `registration-cert-drafter`）
   - 核验声明内容与前三层证据的一致性
5. **执行证据完整性检查**：逐项过检查清单，标记通过/失败/不适用。
6. **识别证据缺口**：列出所有未通过的检查项，标注严重程度和补证建议。
7. **生成证据材料目录**：将所有证据整理为结构化清单，包含文件名、类型、状态、核验方式。
8. **评估登记就绪度**：根据完整性评分和缺口严重程度，给出登记就绪评估。
9. **提出后续行动建议**：引用相关的关联技能。
10. 输出完整证据链报告。当需要生成登记证书草稿时，通过 `foundation/handoff-router` 路由至 `registration-cert-drafter`。

## 常见问法与处理策略

| 问法 | 处理策略 |
|------|----------|
| "证据丢了怎么办？" | 根据缺口类型给出补证建议。合同丢失可联系交易对手方补签或出具确认函；日志丢失需评估能否从备份中恢复。无法补证的需在报告中如实标注并在登记时向审查机构说明。 |
| "所有证据必须原件吗？" | 优先使用原件。如原件确实不可得，副本需经公证或可靠电子签名认证。区块链存证的电子证据可视为原件等效形式。 |
| "证据链和审核机构要求不一致" | 不同登记机构（如各地数据交易所）对证据形式有不同要求。应优先参照目标登记机构的材料清单调整证据收集策略。 |
| "我是买方，卖方不愿提供来源证据" | 这是重大交易风险。建议1) 在买卖合同中约定来源合法性保证条款和违约责任；2) 要求卖方签署权属承诺函；3) 评估交易风险后决定是否继续。 |
| "数据经过多手流转，原始来源证据掉了" | 多手流转场景中，每一手都应保留前手的权属证明。如果中间环节断裂，从最近的可靠来源开始构建，并在报告中注明历史追溯的限制。 |
| "同一个数据资产在不同交易所登记证据要求不同" | 建议以最高标准准备证据包（覆盖所有常见交易所的要求），以降低重复收集的成本。调用本技能时指定 `registration_authority` 以获取针对性建议。 |
| "我们的数据是 AI 生成的，还需要来源合法性证据吗？" | 需要。AI 生成数据的来源合法性不仅包括模型训练数据的合法性，还包括生成过程的合法使用授权。AI 生成数据的三权分析有其特殊性，需结合 AI 服务协议判定。 |
