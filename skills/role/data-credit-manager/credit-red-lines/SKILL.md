# 🚩 credit-red-lines

数据信用红线清单。由数据信用管理机构（`dam-10`）制定和维护，列出数据交易信用评价中的**一票否决项**。在每次授信审批前，必须逐项核查被评对象是否触发红线；任一条红线触发，则**无条件拒绝授信**，不再进入后续评分或评级流程。

> 风险等级：**高**。红线的判断直接影响授信决策，误判可能导致合规风险或经济损失。红线判断须有充分证据，不可仅凭怀疑。

## When to Use

- 新主题/数据产品**首次申请**信用评分或授信时。
- 现有已授信对象**发生重大事件**（涉诉、违规、安全事故等）。
- 每季度**定期审查**已授信对象的红线状态。
- 信用政策修订时，需同步更新红线清单。

## 工作原则

1. **刚性执行**：红线触发的后果是一票否决，无例外、无豁免。
2. **证据确凿**：红线触发须基于可验证的事实证据，禁止主观推定或猜测性判断。
3. **分类管理**：按造假、违规、出境、安全、诚信五类组织清单，清晰可查。
4. **持续更新**：随法律法规变化和行业实践演进定期修订红线清单。
5. **保密必要**：红线审查过程和发现不得向被评方及其他无权限人员透露。

## 一、造假类红线

涉及**数据造假、指标造假、业绩造假**的行为，直接触发一票否决。

| 红线编号 | 红线名称 | 触发条件 | 证据要求 | 法律依据 |
|----------|----------|----------|----------|----------|
| F-01 | 数据造假 | 被评方提供的数据量、数据质量指标存在虚构、篡改或严重虚报（差异 > 20%） | 独立核验报告、数据对账记录 | 《数据安全法》第29条 |
| F-02 | 指标造假 | 被评方提交的财务报表、经营指标、技术指标等存在虚假陈述 | 审计报告、工商登记对比、第三方验证 | 《会计法》第43条 |
| F-03 | 身份造假 | 被评方虚报或冒用营业执照、资质证书、知识产权证书等 | 官方查询结果（工商、知产局） | 《行政许可法》 |
| F-04 | 交易记录造假 | 被评方提供的交易历史、履约记录经过篡改或伪造 | 交易平台原始记录对账 | 交易平台规则 |
| F-05 | 重复造假 | 被评方历史上曾被认定有过造假行为且超过整改期后仍有类似行为 | 历史违规档案 | - |

## 二、来源违规类红线

涉及**数据来源不合法、采集不合规、处理违规**的行为。

| 红线编号 | 红线名称 | 触发条件 | 证据要求 | 法律依据 |
|----------|----------|----------|----------|----------|
| S-01 | 来源违法 | 数据来源于窃取、侵入、破坏计算机信息系统等违法行为 | 公安机关立案决定书/判决书 | 《刑法》第285-286条 |
| S-02 | 采集违规 | 未经同意采集个人信息或超范围采集个人敏感信息 | 行政处罚决定书/用户投诉证据 | 《个人信息保护法》第13-17条 |
| S-03 | 爬虫违规 | 违反 robots 协议、突破反爬措施、被抓取方明确拒绝仍持续采集 | 技术分析报告、律师函、法院禁令 | 《反不正当竞争法》 |
| S-04 | 授权缺失 | 数据加工、转让、许可使用无有效授权文件 | 合同审查、授权的链条审查 | 《数据安全法》第33条 |
| S-05 | 超范围使用 | 数据实际使用场景超出用户授权范围或合同约定用途 | 使用范围 vs. 授权范围对比 | 《个人信息保护法》第6条 |

## 三、出境违规类红线

涉及**数据跨境传输未申报、敏感数据违规出境**的行为。

| 红线编号 | 红线名称 | 触发条件 | 证据要求 | 法律依据 |
|----------|----------|----------|----------|----------|
| C-01 | 未申报出境 | 重要数据/个人信息出境但未向网信部门申报数据出境安全评估 | 安全评估申报回执缺失 | 《数据出境安全评估办法》 |
| C-02 | 违规出境 | 国家核心数据、禁止出口数据违规向境外提供 | 网信办/国安部门的认定或通报 | 《数据安全法》第36条 |
| C-03 | 规避监管 | 通过技术手段（加密/拆分/伪装）规避出境监管 | 监管部门认定文件 | 《数据安全法》 |
| C-04 | 出境后失控 | 数据出境后未按承诺用途使用或未落实安全保障措施 | 境外接收方审计报告 | 《个人信息保护法》第38条 |

## 四、安全违规类红线

涉及**重大数据泄露、PII严重违规、安全审计不合格**的事件。

| 红线编号 | 红线名称 | 触发条件 | 证据要求 | 法律依据 |
|----------|----------|----------|----------|----------|
| A-01 | 重大泄露 | 被评方发生影响 > 100万个人信息或重要数据的泄露事件 | 事件通报、主管部门处罚 | 《个人信息保护法》第57条 |
| A-02 | PII严重违规 | 因违反PII相关规定被网信部门处以5000万以上或上年营业额5%以上的罚款 | 行政处罚决定书 | 《个人信息保护法》第66条 |
| A-03 | 安全审计不合格 | 连续两次安全等级保护测评/数据安全审计结论为不合格 | 审计/评测报告 | 《数据安全法》第30条 |
| A-04 | 未落实整改 | 被监管部门责令整改但逾期未完成整改 | 整改通知 + 复查不合格报告 | 《数据安全法》第44条 |

## 五、诚信类红线

涉及**列入失信名单、重大涉诉、法人犯罪**的事项。

| 红线编号 | 红线名称 | 触发条件 | 证据要求 | 法律依据 |
|----------|----------|----------|----------|----------|
| I-01 | 失信被执行人 | 被评方或其法定代表人被列入失信被执行人名单 | 中国执行信息公开网查询记录 | 最高法失信名单制度 |
| I-02 | 重大涉诉 | 被评方涉及金额 > 净资产20% 的重大未决诉讼且存在不利判决可能 | 法院立案通知书/公告 | - |
| I-03 | 法人犯罪 | 法定代表人或实际控制人因经济犯罪/数据犯罪被刑事立案 | 公安机关立案决定书/检察院起诉书 | 《刑法》 |
| I-04 | 行业禁入 | 被评方或其主要负责人被行业主管部门处以市场禁入 | 行业主管部门公告 | - |
| I-05 | 破产清算 | 被评方进入破产程序 | 法院裁定书 | 《企业破产法》 |

## 红线判定流程

```
开始
  ↓
逐条红线扫描（按 F → S → C → A → I 顺序）
  ↓
触发任一条？
  ├─ 是 → 判定为一票否决 → 终止评分流程 → 记录红线触发日志
  └─ 否 → 继续向前扫描，直至全部完成
              ↓
          未触发任何红线
              ↓
          通过红线审查 → 进入后续评分流程（credit-scorecard-builder）
```

## Inputs

| 字段 | 类型 | 必填 | 说明 | 示例 |
|------|------|------|------|------|
| `subject_id` | string | 是 | 被评对象唯一标识 | `"ENT-2024-00123"` |
| `subject_type` | enum | 是 | 被评对象类型 | `"数据交易主体"` / `"数据产品"` |
| `check_items` | array | 否 | 指定检查的红线类别，默认全部 | `["造假类", "安全违规类"]` |
| `evidence_sources` | object | 否 | 预先准备的证据来源清单 | 加速审查流程 |

## Outputs

| 字段 | 类型 | 说明 |
|------|------|------|
| `red_line_result` | object | 红线审查结果（通过/否决） |
| `triggered_red_lines` | array | 触发的红线清单（含编号、名称、触发时间、证据） |
| `all_checks` | array | 全部25条红线的逐条检查结果 |
| `recommendation` | string | 一票否决或通过继续的建议 |
| `evidence_log` | object | 每个触发项的证据链接（文件编号） |

## Steps

1. **确认审查对象**：核实 subject_id 和 subject_type，确保审查对象正确。
2. **收集证据素材**：从工商、司法、安全、监管等渠道收集被评方的相关记录。
3. **扫描造假类红线（F-01 ~ F-05）**：逐条检查，记录证据和结论。
4. **扫描来源违规类红线（S-01 ~ S-05）**：逐条检查，记录证据和结论。
5. **扫描出境违规类红线（C-01 ~ C-04）**：逐条检查，记录证据和结论。
6. **扫描安全违规类红线（A-01 ~ A-04）**：逐条检查，记录证据和结论。
7. **扫描诚信类红线（I-01 ~ I-05）**：逐条检查，记录证据和结论。
8. **判定**：任一触发 → 一票否决，结束；全部无触发 → 通过，进入下一流程。
9. **出具红线审查报告**：汇总检查结果，以结构化形式输出。
10. **标注标准溯源**：每条红线注明法律依据条文编号。

## 常见问法与处理策略

| 问法 | 处理策略 | 注意事项 |
|------|----------|----------|
| "有一个红线好像触发了，但不确定" | 证据不充分时不可判定触发；标记为"存疑待查"，补充证据后再判 | 宁可放行待查也不要在证据不足时判定一票否决 |
| "红线触发后有没有申诉通道" | 被评方可提供反证材料申请复核；复核由 `dam-10` 指定未参与原判定的人员进行 | 申诉时限：收到否决通知后15个工作日内 |
| "被评方已经整改了，还算不算触发" | 视红线类型而定：造假类/I类事件定型后不可消除；S/A/C类的整改完成并经验收可在12个月后移出 | 造假类红线不可清洗 |
| "如果是关联公司触发了红线，算不算" | 若关联公司与本次被评对象存在实质性控制关系（持股 > 50%），延伸审查 | 需明确关联关系 |
| "红线清单多久更新一次" | 定期每半年评审修订；法律法规更新后30天内临时修订 | 更新后通知所有 `dam-10` 成员 |
