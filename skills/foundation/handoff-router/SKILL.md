# 🔀 handoff-router

## When to Use

当出现以下场景时调用本技能：

- 当前用户的问题**跨出本岗位职责边界**，需要转交给其他岗位 agent
- 多步骤任务中某个步骤属于其他岗位的职责
- 用户明确要求"找负责 XX 的人""这个问题归谁管"
- 当前 agent 能力不足以完整解决用户问题，需要协同

本技能是**全员挂载的协作路由能力**，确保跨岗位问题能被正确路由到对应的专业岗位。

## 工作原则

1. **职责边界清晰**：每个岗位的职责以 T/MIITEC 025-2024 的岗位定义为边界，不越俎代庖
2. **转交不等于推诿**：转交时必须附上当前已完成的工作和上下文摘要，不让目标 agent 从零开始
3. **一次只转一个主目标**：如涉及多岗位，标出主转交目标和抄送目标
4. **用户知情**：转交前必须告知用户转交原因和目标，征得用户同意

## Inputs

- **当前 agent id**（string）：当前正在服务的岗位（如 `dam-03`）
- **用户问题**（string）：超出当前岗位职责的问题描述
- **已完成工作**（string）：当前 agent 已完成的工作摘要
- **上下文信息**（object）：相关数据资产信息、用户偏好、业务背景

## Steps

### 步骤一：识别越界信号

1. 分析用户问题，判断是否超出当前岗位的核心职责（对照 T/MIITEC 025-2024 岗位定义）
2. 常见越界信号：
   - 用户问题涉及不同专业领域（如确权师被问估值）
   - 需要不同岗位的专业技能（如安全师被问会计分录）
   - 任务流程进入下一阶段（如规划师完成后转运营师）

### 步骤二：匹配目标岗位

1. 在岗位转交映射表中查找匹配的目标岗位
2. 如涉及多岗位，确定主转交目标（负责核心问题的岗位）和抄送目标（需要知会的岗位）
3. 标准溯源标注：`[T/MIITEC 025-2024 §5.X.X]`

### 步骤三：构造上下文摘要

1. 汇总当前 agent 已完成的工作（结论、数据资产状态、已确认的信息）
2. 提取与目标岗位相关的关键信息（如已确权 → 估值需要的权属信息已就绪）
3. 记录用户的原始问题与偏好

### 步骤四：输出转交建议卡

1. 生成结构化转交建议卡（from/to/reason/context/skills/urgency）
2. 向用户说明转交原因，征得同意
3. 建议目标岗位应加载的技能
4. 如为紧急事项（如数据泄露），标注 urgency 为 `urgent`
